# Recall Rogue — Quiz & Deck System Redesign

> **Document Type:** Redesign specification. This document describes changes to the existing game design (v2).
> It is intended to be ingested by development agents alongside the current Game Design Document.
> Where this document conflicts with the GDD, this document takes precedence.

---

## 1. Problem Statement

### The Core Flaw: Cross-Domain Answer Leakage

The current system mixes facts from multiple knowledge domains within a single run pool. This creates a fatal educational flaw: **answers become identifiable by format rather than knowledge.**

When a run contains geography, history, and chemistry facts, the player learns that:
- "Cleopatra" is always the history answer
- "Helium" is always the chemistry answer
- "Mount Everest" is always the geography answer

This is the *process of elimination effect* — the player memorizes which answer belongs to which domain rather than learning the actual content. After 3-4 encounters with the same pool, quiz difficulty collapses regardless of timer pressure or distractor count.

### Secondary Flaw: Fact-Card Binding Creates Free Hits

The current system binds facts to card slots at draw time. Once a player masters a fact, the card carrying it becomes a guaranteed power play every time it appears. This inverts the intended difficulty curve — the more you play, the EASIER the quizzes become, reducing both educational value and gameplay tension.

### Secondary Flaw: LLM-Generated Distractors Are Inherently Weak

Pre-generated distractors lack context about what the player actually confuses. A distractor pool generated by an LLM might include plausible-looking wrong answers, but they aren't calibrated to the player's actual knowledge gaps. The system cannot adapt distractors to individual confusion patterns.

---

## 2. Solution Overview

Three interconnected changes:

1. **Domain-Focused Curated Decks** — Each run uses a single curated deck (e.g., "US Presidents," "Japanese N5 Vocabulary," "Chemical Elements"). All facts and distractors come from within this deck.

2. **Dynamic Fact Assignment at Charge Time** — Facts are not bound to card slots. When a player charges a card, the system selects a fact from the deck pool in real time, weighted by in-run performance data.

3. **Pool-Based Adaptive Distractors** — Distractors are drawn from the deck's own fact pool, weighted by the player's confusion matrix (which answers they've previously confused with each other).

## 3. Curated Deck System

### 3.1 Deck Structure

Each curated deck is a self-contained educational unit designed for a specific knowledge domain or sub-domain.

```typescript
interface CuratedDeck {
  id: string;                          // e.g., "us_presidents", "japanese_n5_vocab"
  name: string;                        // Display name
  domain: string;                      // Parent domain for knowledge tracking
  subDomain?: string;                  // Optional sub-domain
  description: string;                 // Player-facing description
  minimumFacts: number;                // Minimum facts required (default: 30)
  targetFacts: number;                 // Ideal fact count (default: 50)
  facts: DeckFact[];                   // The fact pool
  answerTypePools: AnswerTypePool[];   // Sub-pools grouped by answer format
  synonymGroups: SynonymGroup[];       // Groups of facts whose answers overlap (never distract each other)
  chainThemes: ChainTheme[];           // Deck-specific chain type definitions
  questionTemplates: QuestionTemplate[]; // Deck-specific question formats
  difficultyTiers: DifficultyTier[];   // Internal difficulty groupings
}

interface DeckFact {
  id: string;
  correctAnswer: string;               // Primary correct answer
  acceptableAlternatives: string[];    // Other accepted answers (synonyms, alternate phrasings)
  synonymGroupId?: string;             // Facts sharing a group NEVER distract each other
  chainThemeId: number;                // Which chain theme this fact belongs to
  answerTypePoolId: string;            // Which answer pool for distractor sourcing
  difficulty: number;                  // 1-5 internal difficulty
  funScore: number;                    // 1-10 engagement rating
  // ... additional fields per deck type (e.g., japanese_word, reading, explanation)
}

interface SynonymGroup {
  id: string;                          // e.g., "pretty_beautiful_group"
  factIds: string[];                   // All facts in this group
  reason: string;                      // Why grouped (for debugging/auditing)
}
```

### 3.2 Answer Type Pools

Facts within a deck are grouped into **answer type pools** — sets of facts whose answers are the same format and therefore can serve as distractors for each other.

```typescript
interface AnswerTypePool {
  id: string;                    // e.g., "president_names", "inauguration_years", "party_names"
  label: string;                 // Human-readable label
  answerFormat: string;          // "name" | "year" | "number" | "term" | "place" | "word" | etc.
  factIds: string[];             // Facts belonging to this pool
  minimumSize: number;           // Minimum 5 facts per pool (1 correct + 4 distractors)
}
```

**Minimum pool size: 5 facts.** With fewer than 5, the player will pattern-match distractors within 2-3 encounters. If a sub-pool has fewer than 5 facts, it must be merged with a related pool or use the existing bracket-number generation system for numeric answers.

**Example — US Presidents deck:**

| Pool ID | Answer Format | Example Answers | Count |
|---------|--------------|-----------------|-------|
| `president_names` | Name | Washington, Lincoln, Roosevelt, Obama... | 46 |
| `inauguration_years` | Year (bracket) | {1789}, {1861}, {1933}, {2009}... | 46 |
| `party_names` | Term | Federalist, Whig, Republican, Democrat... | ~8 |
| `home_states` | Place | Virginia, Illinois, New York, Hawaii... | ~30 |
| `ordinal_numbers` | Number (bracket) | {1}, {16}, {32}, {44}... | 46 |

**Example — Japanese N5 Vocabulary deck:**

| Pool ID | Answer Format | Example Answers | Count |
|---------|--------------|-----------------|-------|
| `english_meanings` | English word/phrase | to eat, to drink, delicious, big... | 822 |
| `japanese_words` | Japanese word | 食べる, 飲む, おいしい, 大きい... | 822 |
| `reading_hiragana` | Hiragana | たべる, のむ, おいしい... | 822 |

### 3.3 Bracket Number System (Preserved)

The existing system for numeric answers using brackets (`{8848}`, `{1789}`) is preserved and extended. When a fact's answer is numeric, the system generates plausible wrong numbers at runtime rather than pulling from the answer type pool.

This applies to: years, measurements, counts, distances, populations, dates, ordinal numbers, and any other quantitative answer.

Bracket-number facts still belong to an answer type pool (for organizational purposes) but their distractors are generated, not pooled.

### 3.4 Deck Selection at Run Start

At run start, the player enters the **Dungeon Selection Screen** — a single unified screen with two mode tabs:

1. **Trivia Dungeon** — casual mode. Pick one or more domains and/or subdomains from the general trivia fact pool (`facts.db`, `knowledge-*.json`). Multi-select supported. Facts are standalone entries, no curated structure.
2. **Study Temple** — focused study mode. Browse curated decks organized by domain (left sidebar). Pick a specific deck, optionally focus on a sub-deck (e.g., Japanese N3 -> Kanji only). All facts come from that deck's dedicated fact file (`data/decks/*.json`).

**CRITICAL: These two modes use COMPLETELY SEPARATE fact pools.** Trivia Dungeon draws from the general trivia database. Study Temple draws from curated deck JSON files. There is zero overlap between the fact pools. Both can cover the same topics but use entirely different fact entries.

**Vocabulary content (all languages) lives EXCLUSIVELY in Study Temple** as curated decks with sub-decks (Vocabulary, Kanji, Grammar). Language facts are not available in Trivia Dungeon.

The screen remembers the player's last selected mode and configuration for quick re-entry. See AR-244 for full UX specification.

**Multiple deck composition (future consideration):** For advanced players who want variety, a "Mixed Deck" option could combine 2-3 curated decks — but only if their answer type pools don't overlap in confusing ways. This is NOT for initial implementation.

### 3.5 Deck-Specific Chain Themes

Instead of the generic 6 chain types (Obsidian, Crimson, Azure, Amber, Violet, Jade), curated decks define their own **thematic chain types** that map to meaningful sub-groupings within the domain.

```typescript
interface ChainTheme {
  id: number;                    // 0-N (minimum 3 themes per deck, no upper limit)
  name: string;                  // e.g., "Founding Fathers", "Civil War Era"
  color: string;                 // Hex color
  icon: string;                  // SVG icon path
  factSubset: string[];          // Fact IDs that belong to this theme (minimum 8 per theme)
}
```

**CURRENT IMPLEMENTATION: Generic chain slots only.** For initial deck builds, facts are assigned to generic chain slot indices (0-5) distributed evenly. The 6 default chain types (Obsidian, Crimson, Azure, Amber, Violet, Jade) are used for all decks — both knowledge and vocabulary. Named thematic chain groupings (e.g., "Founding Fathers", "Civil War Era") are a future enhancement for replayability. The in-game chaining mechanic works correctly with generic slots.

**Minimum 3 chain themes per deck.** No upper limit — a large deck like the Periodic Table might have 8+ themes (Alkali Metals, Noble Gases, Transition Metals, Halogens, etc.). Each run selects 3 of the deck's themes (deterministically from seed), same as the current system. This means a deck with exactly 3 themes always uses all of them, while a deck with 8 themes has different 3-theme combinations per seed — adding replayability.

**Example — US Presidents deck:**

| Chain ID | Theme Name | Color | Facts |
|----------|-----------|-------|-------|
| 0 | Founding Fathers | #546E7A | Washington, Adams, Jefferson, Madison, Monroe |
| 1 | Civil War Era | #EF5350 | Lincoln, Johnson, Grant, Hayes, Garfield |
| 2 | Progressive Era | #42A5F5 | T. Roosevelt, Taft, Wilson, Harding, Coolidge |
| 3 | Modern Era | #FFA726 | FDR, Truman, Eisenhower, JFK, LBJ |
| 4 | Late 20th Century | #AB47BC | Nixon, Ford, Carter, Reagan, Bush Sr. |
| 5 | 21st Century | #26A69A | Clinton, Bush Jr., Obama, Trump, Biden |

**Example — Japanese N5 Vocabulary deck: NO chain themes.**

Vocabulary decks skip thematic chain types. The semantic groupings in vocabulary (actions, adjectives, nouns) don't create meaningful strategic chain decisions — "I'll chain two adjective cards" is not an interesting choice. Instead, vocabulary deck cards are assigned **generic chain types** (using the 6 default colors: Obsidian, Crimson, Azure, Amber, Violet, Jade) distributed evenly across the deck by the seed. 3 of the 6 are selected per run as usual.

When a vocabulary card is charged, the fact is pulled from the **entire deck pool** (not a theme subset), weighted by in-run FSRS. Chains still work mechanically — matching chain colors still give multipliers — but the chain color carries no educational meaning. It's purely a mechanical/strategic layer.

**This distinction applies to ALL vocabulary decks across all languages.** Knowledge decks (Presidents, Periodic Table, etc.) use thematic chains. Vocabulary decks use generic chains.

**Run chain selection:** Each run selects 3 of the deck's chain themes (deterministically from seed). Cards are assigned to these 3 themes. When a card is charged, the fact is pulled from that card's chain theme's fact subset. Decks with exactly 3 themes always use all of them; decks with more themes get different 3-theme combinations per seed.

**This means cards ARE bound to a sub-pool (chain theme), but NOT to a specific fact.** The card shows "Civil War Era" chain coloring, so you know the quiz will be about a Civil War era president — but you don't know WHICH one until you charge. **Exception: vocabulary decks** — cards have generic chain colors with no thematic sub-pool binding; facts are drawn from the full deck pool.

### 3.6 Card Visual Identity Under New System

**GDD Change: Cards no longer display facts on the card face.** The current GDD describes cards showing question text, difficulty stars, and domain color. All of this is removed. Cards are purely mechanical objects with chain theme identity.

Cards in hand show:
- **Mechanic** (Strike, Block, Hex, etc.) — unchanged
- **AP cost** — unchanged
- **Chain theme color + icon** — tells you the sub-topic pool
- **Chain theme name** — small label (e.g., "Founding Fathers")
- **Mastery level** (0-5) — per card slot, reflects quiz performance with this card
- **NO specific fact, NO question text, NO difficulty stars** — the quiz content is completely hidden until charge

This creates a strategic layer: "I have two Founding Fathers cards and one Civil War card. If I charge the Founding Fathers cards consecutively, they'll chain. But I need to answer correctly on Founding Fathers content."

The card's visual identity comes from its mechanic type (border color) and chain theme (banner color + icon + name). This is sufficient for all strategic decisions: which cards to play, which to chain, which to charge.

## 4. Dynamic Fact Assignment

### 4.1 When Facts Are Assigned

Facts are assigned to cards **at the moment of charge**, not at draw time.

**Current flow (replaced):**
```
Draw hand → facts assigned to cards → player sees facts → decides to charge → quiz with pre-assigned fact
```

**New flow:**
```
Draw hand → cards show mechanics + chain theme only → player charges a card →
system selects fact from chain theme's sub-pool → quiz with dynamically selected fact
```

### 4.2 Fact Selection Algorithm

When a card is charged, the system selects a fact from the intersection of:
1. The card's chain theme fact subset (for knowledge decks) OR the full deck pool (for vocabulary decks with generic chain types)
2. The deck's available fact pool (minus cooldown)

Selection is weighted by **in-run FSRS state:**

```typescript
function selectFactForCharge(
  factPool: string[],           // theme subset (knowledge) or full pool (vocabulary)
  inRunFSRS: Map<string, InRunFactState>,
  cooldownSet: Set<string>,
  cardMasteryLevel: number
): DeckFact {
  const available = factPool.filter(id => !cooldownSet.has(id));

  // Weight by in-run performance
  const weighted = available.map(id => {
    const state = inRunFSRS.get(id);
    let weight = 1.0;

    if (!state) {
      // Never seen this run — moderate priority
      weight = 1.5;
    } else if (state.wrongCount > state.correctCount) {
      // Struggling with this fact — HIGH priority
      weight = 3.0;
    } else if (state.correctCount >= 2 && state.wrongCount === 0) {
      // Known well this run — LOW priority (hot streak suppression)
      weight = 0.2;
    } else if (state.correctCount === 1) {
      // Seen once, got right — moderate-low
      weight = 0.7;
    }

    // Adjust for card mastery: higher mastery = prefer harder facts
    if (cardMasteryLevel >= 3) {
      // At high mastery, boost weight for facts the player struggles with
      weight *= state?.wrongCount > 0 ? 1.5 : 0.8;
    }

    return { id, weight };
  });

  return weightedRandomSelect(weighted, runSeed);
}
```

### 4.3 In-Run FSRS (Lightweight Session Tracker)

A lightweight per-run knowledge state that tracks session performance. Separate from global FSRS.

```typescript
interface InRunFactState {
  factId: string;
  correctCount: number;         // Times answered correctly this run
  wrongCount: number;           // Times answered wrong this run
  lastSeenEncounter: number;    // Encounter number last seen
  confusedWith: string[];       // Fact IDs this was confused with (wrong answer selections)
  averageResponseTimeMs: number;
  streak: number;               // Current consecutive correct (resets on wrong)
}
```

**Initialization:** At run start, in-run state is seeded from global FSRS:
- Facts with low global stability (< 2 days) start with `wrongCount: 1` (treat as if already struggling)
- Facts with high global stability (> 30 days) start with `correctCount: 1` (treat as somewhat known)
- Everything else starts fresh

**Updates:** Every charge result (correct or wrong) updates both in-run state AND global FSRS state.

### 4.4 Confusion Matrix (Adaptive Distractors)

The system tracks which facts the player confuses with each other:

```typescript
interface ConfusionEntry {
  targetFactId: string;     // The correct answer
  confusedFactId: string;   // What the player chose instead
  count: number;            // How many times this confusion occurred
}
```

When the player answers wrong, the system records: "When asked about Madison, the player chose Monroe." This confusion pair persists across runs (stored in global player data).

**Distractor selection priority (replaces LLM-generated distractors):**

1. **Synonym group exclusion (MANDATORY — checked first):** Any fact in the same `synonymGroupId` as the correct fact is EXCLUDED from the candidate pool entirely. This prevents unfair questions where multiple answers are semantically correct.
2. **Known confusions first:** Facts the player has previously confused with the correct answer (from confusion matrix). These are the hardest, most educational distractors.
3. **In-run struggles:** Facts the player has gotten wrong this run (from in-run FSRS). Plausible because the player is actively uncertain about them.
4. **Same answer-type pool, similar difficulty:** Other facts from the same answer type pool at similar difficulty rating.
5. **Same answer-type pool, any difficulty:** Remaining facts from the pool, ordered by similarity to the correct answer.

```typescript
function selectDistractors(
  correctFact: DeckFact,
  answerPool: AnswerTypePool,
  synonymGroups: SynonymGroup[],
  confusionMatrix: ConfusionEntry[],
  inRunFSRS: Map<string, InRunFactState>,
  count: number,             // 2 at mastery 0, 3 at mastery 1+
  cardMasteryLevel: number
): DeckFact[] {
  // Step 0: Build synonym exclusion set (MANDATORY)
  const synonymExcludeIds = new Set<string>();
  if (correctFact.synonymGroupId) {
    const group = synonymGroups.find(g => g.id === correctFact.synonymGroupId);
    if (group) group.factIds.forEach(id => synonymExcludeIds.add(id));
  }

  const candidates: ScoredCandidate[] = [];

  for (const factId of answerPool.factIds) {
    if (factId === correctFact.id) continue;
    if (synonymExcludeIds.has(factId)) continue;  // Never distract with synonyms

    let score = 1.0;

    // Known confusion — highest priority
    const confusion = confusionMatrix.find(
      c => c.targetFactId === correctFact.id && c.confusedFactId === factId
    );
    if (confusion) score += 10.0 * confusion.count;

    // Reverse confusion (player confused THIS fact with the correct one)
    const reverseConfusion = confusionMatrix.find(
      c => c.targetFactId === factId && c.confusedFactId === correctFact.id
    );
    if (reverseConfusion) score += 5.0 * reverseConfusion.count;

    // In-run struggle
    const runState = inRunFSRS.get(factId);
    if (runState && runState.wrongCount > 0) score += 3.0;

    // Similar difficulty band (±1)
    const diffDelta = Math.abs(correctFact.difficulty - getDifficulty(factId));
    if (diffDelta <= 1) score += 2.0;

    // At higher card mastery, strongly prefer confusable distractors
    if (cardMasteryLevel >= 3) score *= 1.5;

    candidates.push({ factId, score });
  }

  // Sort by score descending, take top N
  candidates.sort((a, b) => b.score - a.score);
  return candidates.slice(0, count).map(c => getFact(c.factId));
}
```

### 4.5 Mastery-Driven Difficulty Scaling

Card mastery level (0-5) now controls quiz difficulty rather than acting as a flat power multiplier on a memorized fact:

| Mastery Level | Distractor Count | Question Variant | Distractor Source Priority |
|---------------|-----------------|------------------|---------------------------|
| 0 | 2 | Easiest available for deck | Random from pool |
| 1 | 3 | Standard | Pool + some confusion-based |
| 2 | 3 | Standard + reverse variants | Confusion-weighted |
| 3 | 4 | Harder variants unlocked | Heavily confusion-weighted |
| 4 | 4 | Hardest available | Maximum confusion-weighting |
| 5 | 4 | Hardest + closest distractors | Known confusions + closest similarity |

**Timer note:** There is no visible quiz timer in the game UI. An invisible internal timer tracks response speed for two purposes: (1) certain relics use response time as a trigger (e.g., Quicksilver Quill rewards sub-2s answers), and (2) the Daily Expedition leaderboard uses response time as a scoring factor. Mastery level does NOT affect the timer. Difficulty scaling is purely through question variant and distractor quality — not artificial time pressure.

**The reward loop:** Higher mastery = higher card power multiplier (unchanged from current system). But the questions get proportionally harder. So mastery 5 gives you a massive multiplier IF you can answer the hardest variant with the most confusable distractors. Knowledge IS power, but only genuine knowledge.

**Fact difficulty also scales with mastery:** At mastery 0, the system prefers to select easier facts from the chain theme pool. At mastery 4-5, it prefers the facts the player struggles with most. This replaces the old "master a fact, get free hits" dynamic with "prove you know the hard stuff to earn the big multiplier."

### 4.6 Synonym Group System (Mandatory for All Decks)

Synonym groups prevent unfair questions where multiple answer options are semantically correct. Facts in the same synonym group are NEVER used as distractors for each other.

**The problem:** If おいしい = "delicious" and another word = "tasty," showing "tasty" as a distractor for おいしい is unfair — both are correct. Similarly, きれい ("pretty/clean/beautiful") and うつくしい ("beautiful/pretty") would create impossible questions if used as each other's distractors.

**The solution: `acceptableAlternatives` intersection at deck build time.** This is entirely programmatic — no LLM required.

```typescript
// Run at deck build time to compute synonym groups
function buildSynonymGroups(facts: DeckFact[]): SynonymGroup[] {
  const groups: Map<string, Set<string>> = new Map();

  for (let i = 0; i < facts.length; i++) {
    const setA = new Set([
      facts[i].correctAnswer.toLowerCase(),
      ...facts[i].acceptableAlternatives.map(a => a.toLowerCase())
    ]);

    for (let j = i + 1; j < facts.length; j++) {
      const setB = new Set([
        facts[j].correctAnswer.toLowerCase(),
        ...facts[j].acceptableAlternatives.map(a => a.toLowerCase())
      ]);

      // Any overlap = same synonym group
      const hasOverlap = [...setA].some(a => setB.has(a));
      if (hasOverlap) {
        // Union-find: merge into same group
        mergeSynonymGroup(groups, facts[i].id, facts[j].id);
      }
    }
  }

  return Array.from(groups.entries()).map(([groupId, factIds]) => ({
    id: groupId,
    factIds: Array.from(factIds),
    reason: 'acceptableAlternatives overlap'
  }));
}
```

**How it works across all languages:**

Every vocabulary fact in every language has:
- A target language word (食べる, 먹다, 吃, comer, manger)
- An English `correctAnswer` ("to eat")
- English `acceptableAlternatives` ("eat", "to consume")

Synonym detection happens entirely on the **English translation layer**, making it language-agnostic. Works identically for Japanese, Korean, Mandarin, Spanish, French, German, Dutch, Czech — every supported language.

**For Forward questions** (target language → English): Distractors are English words from the pool. WordNet synsets (already in the GDD, ~83% coverage for single-word answers) provide additional synonym data. Synonym groups prevent "delicious" and "tasty" from coexisting as separate options.

**For Reverse questions** (English → target language): Distractors are target language words. The system cross-references English translations to detect overlap. If きれい maps to "pretty, clean, beautiful" and うつくしい maps to "beautiful, pretty" — the `acceptableAlternatives` arrays overlap, so they're in the same synonym group.

**For knowledge decks:** Synonym groups are rarer but still needed. Example: a US history deck where "The War Between the States" and "The Civil War" are both correct for the same question — they must be grouped.

**Safety default:** If there's ANY overlap between two facts' acceptable answer sets, group them. Over-grouping (slightly fewer available distractors) is strictly safer than under-grouping (unfair questions).

**The Deck Master skill MUST identify synonym groups during Phase 2 (Architecture) for every deck.** For vocabulary decks this is automated via the intersection algorithm. For knowledge decks it may require manual review of edge cases.

### 4.7 Chain Theme Pool Exhaustion

If a chain theme's available fact pool drops below 3 (due to cooldown), cooldown resets for that theme. This prevents stalling during long boss encounters where 20+ charges might target the same theme.

```
if (availableFactsInTheme < 3) {
  clearCooldownForTheme(chainThemeId);
}
```

If clearing cooldown is still insufficient (theme has fewer than 3 total facts — should not happen with minimum 8 per theme), all theme facts become available regardless of recent-use state.

### 4.8 Non-Combat Quiz Contexts

Shop haggling and Rest Site Study both present quizzes outside of combat. These pull from the **full deck pool** (all chain themes, not theme-specific) since they are not attached to a card with a chain theme.

- **Shop haggling:** 1 question per purchase attempt. Fact selected from full deck pool, weighted by in-run FSRS. Correct = 30% discount. Wrong = full price.
- **Rest Site Study:** 3 questions. Facts selected from full deck pool. Correct answers raise card slot mastery (unchanged mechanic).
- **Boss Quiz Phase:** Questions from full deck pool. Not theme-restricted since the phase is a special event, not a card charge.

All non-combat quizzes still update in-run FSRS and global FSRS. Confusion matrix records apply normally.

## 5. Question Templates (Deck-Specific)

### 5.1 Template System

Each curated deck defines its own question templates rather than using the global variant system. Templates are specific to the deck's subject matter and answer type pools.

```typescript
interface QuestionTemplate {
  id: string;
  answerPoolId: string;           // Which answer type pool this template uses
  questionFormat: string;         // Template string with {placeholders}
  availableFromMastery: number;   // Minimum card mastery to see this template
  difficulty: number;             // 1-5 internal difficulty rating
  reverseCapable: boolean;        // Can this be asked in reverse?
}
```

**Example — US Presidents deck templates:**

### 5.2 Vocabulary Decks — Complete System

**Vocabulary decks are EXCLUSIVELY available in Study Temple mode.** All language learning content (Japanese N5-N1, Korean, Spanish, French, German, Dutch, Czech, etc.) must be structured as curated decks with sub-decks (Vocabulary, Kanji, Grammar where applicable). Language facts are removed from the general trivia fact pool.

Vocabulary decks (all languages) use a standardized template set that works identically across all supported languages. The existing Forward/Reverse/Synonym Pick/Definition Match variants are restructured as deck-specific templates within the new system.

#### Template Mapping (All Languages)

The existing vocabulary variant system maps cleanly to deck-specific templates:

| Old Variant | New Template ID | Answer Pool | Available From Mastery | Notes |
|-------------|----------------|-------------|----------------------|-------|
| Forward | `forward` | `english_meanings` | 0 | "What does 食べる mean?" → English answer |
| Reading | `reading` | `reading_hiragana` | 1 | "What is the reading of 食べる?" → Hiragana. (Japanese/Chinese only) |
| Reverse | `reverse` | `target_language_words` | 2 | "How do you say 'to eat'?" → Target language answer |
| Synonym Pick | `synonym_pick` | `english_meanings` | 3 | "Which word is closest to 食べる?" → English synonym |
| Definition Match | `definition_match` | `english_meanings` | 3 | Shows definition → pick English answer without seeing target word |

**Language-agnostic:** The same template IDs and logic work for Japanese, Korean, Chinese, Spanish, French, German, Dutch, Czech. The `reading` template only applies to languages with separate reading systems (Japanese kanji → hiragana, Chinese hanzi → pinyin). All other templates are universal.

#### Distractors from Deck Pool

All distractors come from the run's deck pool:
- **Forward/Synonym/Definition questions:** Distractors are other English meanings from the `english_meanings` pool
- **Reverse questions:** Distractors are other target language words from the `target_language_words` pool
- **Reading questions:** Distractors are other readings from the `reading_hiragana` (or `reading_pinyin`) pool

The confusion matrix applies naturally: if you've confused 飲む ("to drink") with 食べる ("to eat") three times, 飲む is permanently the #1 distractor whenever 食べる is the answer and vice versa.

#### Process of Elimination — Why It's Acceptable

**Concern:** "If I already know 飲む = 'to drink' and 見る = 'to see', and those appear as distractors for 食べる, I can just pick the unfamiliar English word."

**Why this is fine:** That elimination IS learning. If you can confidently eliminate "to drink" because you genuinely know 飲む = "to drink," you're demonstrating real vocabulary knowledge. The elimination itself is evidence of mastery.

This is only problematic if:
1. The pool is so small that ALL distractors are known mappings (unlikely with N5 at 822 words)
2. The system lacks confusion data to generate hard distractors

The confusion matrix is the real solution. Once the system knows which words you genuinely confuse (built up over multiple runs), those become your distractors — and elimination stops working because you DON'T actually know which is which.

**Early-run behavior:** In the first few encounters of a run, confusion data is sparse and elimination is easier. This is intentional — the player is genuinely learning the mappings, so easier questions are appropriate. As the run progresses, the in-run FSRS + confusion matrix tightens the distractors automatically.

#### Synonym Safety (CRITICAL)

Vocabulary decks have high synonym density. The synonym group system (§4.6) is MANDATORY for all vocabulary decks to prevent unfair questions.

**Common vocabulary synonym situations:**

| Language | Word A | Word B | English Overlap | Must Group? |
|----------|--------|--------|-----------------|-------------|
| Japanese | おいしい | うまい | "delicious", "tasty" | YES |
| Japanese | きれい | うつくしい | "beautiful", "pretty" | YES |
| Japanese | 大きい | でかい | "big", "large" | YES |
| Spanish | bonito | hermoso | "beautiful", "pretty" | YES |
| French | beau | joli | "beautiful", "pretty" | YES |
| Korean | 예쁘다 | 아름답다 | "pretty", "beautiful" | YES |
| Mandarin | 漂亮 | 美丽 | "beautiful", "pretty" | YES |

The `acceptableAlternatives` intersection algorithm (§4.6) catches all of these automatically at deck build time. No manual review or LLM needed.

**Reverse question synonym safety:** For Reverse questions ("How do you say 'pretty'?"), synonym-grouped target language words (きれい and うつくしい) never appear as each other's distractors. The question may also need specificity ("Which word means 'pretty' in the sense of 'clean/tidy'?") — the Deck Master skill should flag these during deck architecture.

#### Per-Language Display Settings

Vocabulary decks support per-language display settings that customize quiz rendering. Settings are scoped to a language and apply to ALL sub-decks of that language. They are accessible from the Study Temple vocabulary tree (cogwheel on language headers) and live during runs (cogwheel on quiz overlay).

Current settings (Japanese only):
- Furigana toggle (hiragana above kanji)
- Romaji toggle (romanized readings)
- Kana-only mode (replace kanji with hiragana)

Future languages may add their own options. Settings are stored via `deckOptionsService.ts` and react immediately in the quiz UI.

#### Vocabulary Deck Restructuring Notes

Existing programmatic vocabulary data (Japanese 13,073 facts, Korean 11,400, Chinese 13,472, etc.) must be restructured with:
1. `acceptableAlternatives` fully populated for all facts (many already have this)
2. Synonym groups computed via intersection algorithm
3. Answer type pools defined (english_meanings, target_language_words, reading_hiragana/reading_pinyin)
4. Difficulty ratings preserved from existing data
5. Fact IDs preserved where possible to maintain player FSRS data
6. **No chain themes needed** — vocabulary decks use generic chain types (see §3.5)
7. **Vocabulary facts removed from Trivia Dungeon** — language content lives exclusively in Study Temple curated decks

### 5.3 Template Selection at Charge Time

When a fact is selected for a charge, the system also selects which question template to use:

1. Filter templates to those available at the current card mastery level
2. Filter to templates whose answer pool contains the selected fact
3. Weight by: difficulty appropriate to mastery level, variety (don't repeat same template consecutively), in-run template history
4. Select weighted random (seeded)

## 6. Removed Systems

The following systems from the current GDD are removed or substantially modified by this redesign:

### 6.1 Removed: Cursed Card System (§11)

**Reason:** Cursed cards required fact-card binding to have visual identity. With dynamic fact assignment, there is no persistent "cursed fact on a card" to display.

**Replacement:** Wrong answers are handled naturally by:
- Lower multiplier on wrong answer (0.6×/0.7× — unchanged)
- The wrong fact gets higher selection weight in future charges (in-run FSRS)
- The confusion is recorded in the confusion matrix (persistent across runs)
- Card slot does not gain mastery on wrong answer

No separate cursed visual state is needed. The consequence of failure is built into the adaptive system.

### 6.2 Removed: Free First Charge (§2, AR-59.23)

**Reason:** Free First Charge solved "don't punish blind guessing on unknown facts." With curated domain decks the player chose themselves, plus adaptive difficulty that starts with easier facts at mastery 0, this problem is solved by design.

All charges cost their normal AP surcharge. The first charge of a run is not special.

### 6.3 Removed: LLM-Generated Distractors (§29)

**Reason:** Pool-based adaptive distractors are strictly superior. They're contextually accurate, adaptively difficult, and don't require an LLM pipeline.

**The rule "NEVER generate distractors from database pools" in §29 is REVERSED.** Distractors MUST come from the deck's own fact pool (with the exception of bracket-number facts, which still use runtime numeric generation).

### 6.4 Removed: Cross-Domain Run Pools (§23)

**Reason:** The core flaw this redesign addresses. Runs no longer mix domains.

The run pool composition table (§23: 30% primary domain, 25% secondary domain, 45% FSRS review) is replaced by: **100% from the selected curated deck.**

### 6.5 Removed: Fact-Card Binding at Draw Time (§2, AR-93)

**Reason:** Per-draw FSRS shuffling (AR-93) assigned facts to card slots each draw. This is replaced by charge-time assignment.

Cards drawn into hand have NO fact. The fact is selected when the player commits to charging.

### 6.6 Modified: Tier System (§5)

The FSRS-driven tier system (Tier 1/2a/2b/3) for individual facts is **decoupled from combat power.** Tiers still exist in global FSRS for long-term knowledge tracking, but they no longer drive card multipliers or auto-charge behavior.

**Combat power** is now driven by **card slot mastery** (0-5, per run) which reflects session performance.

**Tier 3 auto-charge is removed.** There is no fact-level mastery that eliminates quizzes. Every charge always presents a quiz. At mastery 5, the quiz is the hardest variant — but the multiplier is also the highest.

### 6.7 Modified: Mastery Trial (§5)

Mastery Trials as a fact-level graduation ceremony are removed (no Tier 3 auto-charge to graduate into).

**Possible replacement (future consideration):** A run-level "Mastery Challenge" at the final boss where the hardest facts from the run are presented in rapid succession. This is a climactic test, not a graduation.

### 6.8 Modified: Question Format by Tier (§5)

The tier-based question format table is replaced by mastery-based template selection (see §5.3 above). Question difficulty scales with card slot mastery, not fact FSRS tier.

### 6.9 Clarified: Dynamic Timer System (§2)

**There is no visible quiz timer in the game UI.** The GDD §2 "Dynamic Timer System" table (12s → 9s → 7s → 5s → 4s by floor) describes an **invisible internal timer** only. The player sees no countdown bar, no ticking clock, no time pressure UI.

The invisible timer serves two purposes:
1. **Relic triggers:** Relics like Quicksilver Quill ("answered in under 2 seconds") and Adrenaline Shard ("under 3 seconds") check against it
2. **Leaderboard scoring:** Daily Expedition and competitive runs use response time as a scoring component — faster correct answers score higher

The "Slow Reader mode" setting (+3s) still applies to the invisible timer thresholds. The timer still auto-fizzles cards if no answer is given (preventing AFK stalling), but this timeout is generous and not communicated as time pressure.

**GDD references to timer visuals (red timer bar, amber timer bar, timer-related VFX) should be removed or moved to the invisible timer's internal tracking only.**

## 7. Global FSRS — New Role

### 7.1 What Global FSRS Still Does

- **Long-term knowledge tracking:** Every quiz result (correct/wrong, response time) updates the global FSRS state for that fact
- **Run pool seeding:** Facts with low global stability are flagged as "needs practice" when initializing in-run state
- **Knowledge visualization:** Powers the cross-run knowledge map showing mastery across domains and sub-domains
- **Deck difficulty estimation:** Average global FSRS stability across a deck's facts tells the player "you know 73% of US Presidents well"

### 7.2 What Global FSRS No Longer Does

- Does NOT drive card tier (1/2a/2b/3) for combat power
- Does NOT trigger Mastery Trials
- Does NOT determine auto-charge eligibility
- Does NOT set question format (that's now card mastery + deck templates)

### 7.3 Knowledge Visualization (Future Feature)

Since global FSRS tracks per-fact correctness over time across all runs, and the confusion matrix tracks every wrong-answer pairing permanently, the system can build rich knowledge visualizations:

- **Per-deck mastery map:** "You know 40/46 presidents. You struggle with 1840-1880 era. You consistently confuse the two Adamses."
- **Cross-deck domain view:** "History: 73% mastery. Science: 45%. Languages: 62%."
- **Temporal decay view:** "You knew Chemical Elements well 3 months ago but haven't practiced — estimated current recall: 48%."
- **Confusion graph (surfaced from confusion matrix):** "Your top confusions: Monroe ↔ Madison (confused 7 times), 飲む ↔ 食べる (confused 12 times), Helium ↔ Hydrogen (confused 4 times)." Players can see their exact blind spots and target them.
- **Per-deck confusion heatmap:** Visual grid showing which facts you confuse with which other facts. Denser clusters = areas to focus study.
- **Math mistake patterns:** "You make sign errors 34% of the time on subtraction problems. You forget to distribute 22% of the time."

This becomes a genuinely valuable study tool independent of the game — a strong retention hook. The confusion matrix data is unique to Recall Rogue; no other educational game or flashcard app provides this level of personalized confusion analysis.

## 8. Trivia Dungeon & Daily Expedition

### 8.1 Trivia Dungeon (Separate Game Mode)

A separate mode where instead of a focused curated deck, facts are drawn from a broad domain pool with no repetition within a run. Every fact appears at most ONCE.

| Aspect | Study Temple | Trivia Dungeon |
|--------|--------------|-------------|
| Deck | Curated, 30-50+ facts, one domain | Broad pool, 200+ facts, one or mixed domains |
| Repetition | Facts repeat across encounters (FSRS-weighted) | Each fact appears ONCE per run, never repeated |
| Distractors | Pool-based, confusion-adaptive | Pool-based but NOT confusion-adaptive (not enough repetition to build confusion data) |
| Fact-card binding | Dynamic at charge time | Dynamic at charge time (same) |
| In-run FSRS | Active (drives adaptive selection) | Minimal (just tracks seen/unseen) |
| Learning intent | Deep mastery of focused topic | Broad exposure to many facts |
| Chain themes | Deck-specific themed sub-groups | Generic or domain-based |
| FSRS update | Yes — results update global FSRS | Yes — results still update global FSRS |

**Fact Pool Separation:** Trivia Dungeon uses the general trivia database (`facts.db`, `knowledge-*.json`). Study Temple uses curated deck files (`data/decks/*.json`). These are completely separate pools — no fact IDs overlap. Vocabulary/language facts are exclusive to Study Temple.

**When to use Trivia Dungeon:**
- Casual sessions where variety is more fun than depth
- Exploring a new domain before committing to a curated deck
- Players who find focused repetition tedious

### 8.2 Daily Expedition (Redesigned)

The Daily Expedition uses **Trivia Dungeon mechanics** (no repeat facts, broad pool) with a shared seed so all players face identical questions.

**Per-domain leaderboards:** Each domain has its own Daily Expedition. Players compete to prove they're the best at a specific topic.

| Expedition | Pool Source | Audience |
|------------|-----------|----------|
| Geography Daily | All geography facts | "Best geographer today" |
| History Daily | All history facts | "Best historian today" |
| Science Daily | All science facts | "Best scientist today" |
| Japanese N5 Daily | Japanese N5 vocabulary | "Best Japanese student today" |
| ... (one per major domain) | ... | ... |
| **Ultimate Daily** | **ALL domains mixed** | "Best general knowledge today" |

**Scoring formula:** `score = correctAnswers × speedBonus × depthReached × chainBonus`

Where `speedBonus` is derived from the invisible internal timer — faster correct answers score higher. This is the primary differentiator between players who both answer correctly.

**Rules:**
- One attempt per player per expedition per day
- Same seed = same enemy sequence, same card draws, same fact order for all players
- Timed — the invisible timer's response speed is a major scoring component
- Results submitted to per-domain leaderboard with `metadata.dateKey` (`YYYY-MM-DD`)
- Rewards: participation badge, bonus for top 10%/25%/50% per domain

## 9. Deck Creation Skill (Master Skill Specification)

A single Claude agent skill with three invocable phases for creating high-quality curated decks.

### 9.1 Phase 1: Deck Discovery

**Purpose:** Research what people actually want to learn. Identify high-demand deck topics.

**Inputs:** Optional domain focus (e.g., "History", "Languages", "Science") or open-ended.

**Process:**
1. Search Anki shared deck repositories for download counts and popularity
2. Search Reddit study communities (r/languagelearning, r/medicalschool, r/history, etc.) for recurring study requests
3. Search educational forums and quiz sites for popular topics
4. Analyze trending educational content (YouTube, TikTok edu-creators, podcast topics)
5. Cross-reference with existing game decks to avoid duplicates

**Output:** Ranked list of deck ideas with demand signals:
```
1. "US Presidents" — 50k Anki downloads, frequent Reddit requests, strong sub-grouping potential
2. "Chemical Elements" — 120k Anki downloads, natural periodic table grouping, excellent for chains
3. "World Capitals" — 200k Anki downloads, but may be too shallow for 50-fact deck (196 countries)
4. "Dog Breeds" — niche but passionate community, 30k Anki downloads, visual learning potential
```

### 9.2 Phase 2: Deck Architecture

**Purpose:** For a chosen topic, design the complete pedagogical structure before any facts are generated.

**Inputs:** Topic name, target fact count, any specific constraints.

**Process:**
1. **Deep domain research** — Wikipedia, Wikidata, authoritative sources. Understand the full scope of the topic.
2. **Identify natural answer types** — What kinds of answers exist in this domain? Names, dates, places, terms, categories?
3. **Define answer type pools** — Group potential facts by answer format. Verify each pool has 5+ members.
4. **Design question templates** — For each answer type pool, what question formats make sense? What's easy vs. hard?
5. **Identify common confusions** — What do people typically mix up in this domain? (Washington/Adams/Jefferson for early presidents; noble gases for each other; Monroe/Madison for "James M___" presidents)
6. **Identify synonym groups** — Which answers are semantically interchangeable? For vocabulary: run `acceptableAlternatives` intersection. For knowledge: flag facts with overlapping correct answers (e.g., "Civil War" / "War Between the States"). Ensure synonym-grouped facts never appear as each other's distractors.
7. **Define chain themes** — What are the natural sub-groupings within this topic? Minimum 3, no upper limit. These become chain types. A Periodic Table deck might have 8+ themes; a US Presidents deck has 6; a small vocabulary deck might have exactly 3.
8. **Set difficulty tiers** — Which facts are universally known (easy), commonly known (medium), and obscure (hard)?
9. **Validate structure** — Ensure every answer type pool has ≥5 members (after synonym exclusions), every chain theme has ≥8 facts, total facts ≥ target. Verify no synonym group is so large that it starves the distractor pool.

**Output:** A deck architecture specification:
```yaml
deck_id: us_presidents
name: "US Presidents"
domain: history
sub_domain: american_government
target_facts: 46
answer_type_pools:
  - id: president_names
    format: name
    estimated_count: 46
    min_distractors: 4
  - id: inauguration_years
    format: bracket_number
    estimated_count: 46
    note: "Use bracket system for year generation"
  - id: party_names
    format: term
    estimated_count: 8
    note: "Small pool — merge with era labels if needed"
  - id: home_states
    format: place
    estimated_count: 30
    note: "Some states produced multiple presidents"
chain_themes:
  - id: 0
    name: "Founding Fathers"
    facts: ["washington", "j_adams", "jefferson", "madison", "monroe"]
  - id: 1
    name: "Civil War Era"
    facts: ["lincoln", "a_johnson", "grant", "hayes", "garfield"]
  # ... etc
question_templates:
  - id: name_from_number
    format: "Who was the {ordinal} president of the United States?"
    answer_pool: president_names
    available_from_mastery: 0
    difficulty: 1
  - id: number_from_name
    format: "What number president was {name}?"
    answer_pool: ordinal_numbers
    available_from_mastery: 2
    difficulty: 3
  # ... etc
common_confusions:
  - ["monroe", "madison"]          # James M___
  - ["j_adams", "jq_adams"]       # Father and son
  - ["harrison_wh", "harrison_b"] # Grandfather and grandson
  - ["roosevelt_t", "roosevelt_f"] # Different Roosevelts
  - ["johnson_a", "johnson_l"]     # Different Johnsons
difficulty_tiers:
  easy: ["washington", "lincoln", "obama", "jefferson", "jfk"]
  medium: ["madison", "jackson", "grant", "eisenhower", "reagan"]
  hard: ["fillmore", "pierce", "arthur", "harrison_b", "tyler"]
```

### 9.3 Phase 3: Fact Generation

**Purpose:** Using the architecture from Phase 2, generate the complete fact dataset.

**Inputs:** Deck architecture specification from Phase 2.

**Process:**
1. **Pull structured data** from Wikidata/Wikipedia for each fact
2. **Generate fact entries** conforming to the deck schema, including `acceptableAlternatives` for every fact
3. **Assign facts to answer type pools** and chain themes
4. **Compute synonym groups** by running `acceptableAlternatives` intersection algorithm (§4.6). For vocabulary decks this is fully automated. For knowledge decks, review flagged overlaps manually.
5. **Validate completeness:**
   - Every answer type pool has ≥5 members (after accounting for synonym group exclusions)
   - Every chain theme has ≥8 facts
   - Total facts ≥ minimum deck size
   - No duplicate answers within a pool
   - All question templates have sufficient facts to draw from
   - No synonym group is so large it starves the distractor pool (flag groups > 4 facts)
6. **Assign difficulty ratings** (1-5) based on domain knowledge + deck architecture tiers
7. **Assign fun scores** (1-10) for new player engagement weighting
8. **Quality checks:**
   - Question brevity (max 15 words)
   - Answer brevity (max 5 words / 30 chars)
   - No ambiguous answers (two facts in same pool with identical correct answers that AREN'T in the same synonym group)
   - Bracket numbers are properly formatted
   - Synonym groups correctly computed (spot-check 10% of groups)

**Output:** Complete deck JSON ready for integration.

### 9.4 Skill Invocation

```
/deck-master discover --domain history
/deck-master architect --topic "US Presidents" --target-facts 46
/deck-master generate --architecture us_presidents_arch.yaml
/deck-master full --topic "Chemical Elements" --target-facts 118
```

The `full` command runs all three phases sequentially.

## 11. Resolved Design Decisions

1. **Deck size floor:** Decks must be substantial enough to support pool-based distractors. Minimum 30 facts. Do NOT create ultra-narrow decks like "Noble Gases" (7 elements) — instead create "Periodic Table" (118 elements) or "Chemistry Fundamentals" (60+ facts). Small sub-topics are chain themes within larger decks, not standalone decks.

2. **Multiple domain selection:** Future feature only. Initial implementation supports one curated deck per run. Eventually players may combine 2-3 decks with guardrails against cross-domain answer leakage.

3. **Chain theme count:** Minimum 3 per knowledge deck, no upper limit. Each run selects 3 active themes from the deck's pool. Decks with exactly 3 themes always use all of them. Decks with 8+ themes (like Periodic Table) get different 3-theme combinations per seed, adding replayability.

4. **Daily Expedition:** Per-domain trivia leaderboards, seeded, timed via invisible timer. See §8.2 for full specification.

5. **Card visual identity:** Cards do NOT show facts. This is confirmed and intentional. Chain theme labels + icons (knowledge decks) or generic chain colors (vocabulary decks) provide sufficient strategic information. The GDD sections describing fact text on cards must be updated.

6. **Timer:** No visible timer in the quiz UI. An invisible internal timer tracks response speed for relic triggers and leaderboard scoring only. Mastery difficulty scaling is through question quality and distractor selection, not time pressure.

7. **Vocabulary decks skip chain themes.** Generic chain types (Obsidian, Crimson, etc.) are assigned to cards for mechanical chaining purposes only. Facts are drawn from the full deck pool at charge time, not from a theme subset. Semantic sub-grouping of vocabulary doesn't create meaningful strategic decisions.

8. **Deck unlock model:** All decks free on Steam purchase. Mobile (F2P) gates some decks behind Scholar Pass subscription. Core gameplay decks always free on mobile.

9. **Cross-deck FSRS:** No sharing. Each deck tracks its own FSRS state independently, even if surface content overlaps between decks (e.g., "Tokyo" in World Capitals vs. Japanese Culture).

10. **Chain theme sizing:** Determined per-deck during creation. The Deck Master skill evaluates natural groupings and sizes themes based on the domain's structure. No rigid minimum beyond practical pool exhaustion prevention.

11. **Math decks generate at runtime.** Template rules defined at build time; numbers generated deterministically at charge time from the run seed. Difficulty levels on templates maintain consistent challenge. Common-mistake distractors (sign errors, forgot distribute, etc.) replace pool-based distractors.

12. **Synonym groups are mandatory for all decks.** Computed programmatically via `acceptableAlternatives` intersection — no LLM needed. Works across all languages on the English translation layer. Facts in the same synonym group NEVER distract each other.

### Math Decks — Seed-Based Programmatic Generation

Math decks are unique: questions and answers are **generated from template rules at runtime** rather than stored as static facts. At charge time, the seed + encounter number + charge count produce deterministic coefficients. Same seed = same problem every time. Different seed = different numbers. This gives effectively infinite question variety from a finite set of templates.

**Deck structure:** Instead of individual facts, math decks define **template rules:**

```typescript
interface MathTemplate {
  id: string;                    // e.g., "linear_equation_1"
  category: string;              // "algebra", "geometry", "calculus"
  questionFormat: string;        // "Solve for x: {a}x + {b} = {c}"
  parameters: ParameterRange[];  // [{name: "a", min: 1, max: 10}, {name: "b", ...}]
  solutionFunction: string;      // "(c - b) / a"  — evaluates to correct answer
  commonMistakes: MistakeRule[]; // Distractor generation rules
  difficulty: number;            // 1-5
}

interface MistakeRule {
  label: string;                 // "sign_error", "forgot_distribute", "off_by_one"
  formula: string;               // "(c + b) / a" — the wrong answer a student would get
  weight: number;                // How commonly this mistake occurs
}
```

**Difficulty-level-driven consistency:** Each math template has a difficulty rating (1-5). The system selects templates at charge time weighted by card mastery level, just like knowledge decks select harder facts at higher mastery. This maintains consistent challenge throughout a run:

| Card Mastery | Template Difficulty Pool | Parameter Range |
|-------------|------------------------|-----------------|
| 0-1 | Difficulty 1-2 (basic operations) | Small numbers (1-20) |
| 2-3 | Difficulty 2-3 (multi-step) | Medium numbers (1-100) |
| 4-5 | Difficulty 3-5 (complex, multi-variable) | Large numbers, decimals, negatives |

Within each difficulty level, parameter ranges scale: a difficulty-2 linear equation might use coefficients 1-10, while a difficulty-4 version uses -20 to 50 with fractional solutions. The in-run FSRS tracks which difficulty levels the player gets right/wrong and adjusts template selection accordingly — if you're acing difficulty 3, the system pushes toward 4.

**Distractor generation for math:** Instead of pool-based distractors, math uses **common mistake distractors** — the wrong answers a real student would compute:

| Problem | Correct | Sign Error | Forgot Distribute | Off-by-One |
|---------|---------|-----------|-------------------|------------|
| Solve: 3x + 7 = 22 | 5 | -5 | 10 | 4 or 6 |
| Area of circle, r=5 | 78.54 | 25π ≈ 78.54 (correct) | 31.42 (circumference) | 78 or 79 |
| d/dx of x³ | 3x² | x³ (no change) | 3x (wrong exponent) | 2x² |

**Confusion matrix still applies:** If the player consistently makes sign errors on subtraction problems, the system generates sign-error distractors more frequently. The confusion matrix tracks mistake *types* rather than specific fact confusions.

**Chain themes for math:** Group by operation type or concept area:

| Theme Example (Algebra) | Facts |
|-------------------------|-------|
| Linear Equations | ax + b = c, ax - b = c |
| Quadratic Equations | ax² + bx + c = 0 |
| Systems of Equations | 2-variable systems |
| Inequalities | ax + b > c |

**Seed determinism:** Given the same run seed, the same encounter, and the same charge count, the generated numbers are identical. This enables seeded Daily Expeditions for math where all players solve the same problems.

```
parameterSeed = hash(runSeed, encounterNumber, chargeCount, templateId)
a = seededRandom(parameterSeed, template.parameters["a"].min, template.parameters["a"].max)
b = seededRandom(parameterSeed, template.parameters["b"].min, template.parameters["b"].max)
// ... etc
```

1. **Vocabulary chain themes:** ~~Should vocab decks have semantic chain themes?~~ **No.** Vocabulary decks use generic chain types. Semantic groupings don't create meaningful strategic decisions for vocabulary. Knowledge decks use thematic chains.

2. **Math runtime vs build-time:** ~~Pre-generate or generate at charge time?~~ **Runtime, seed-based.** Template rules are defined at build time; actual numbers are generated deterministically at charge time from the run seed. Difficulty levels on templates maintain consistent challenge throughout a run.

3. **Image question UI:** ~~What layout changes?~~ **Deferred.** Will be designed when the first image-capable deck is built.

4. **Deck unlock model:** ~~Free or gated?~~ **All decks included free on Steam purchase.** Mobile (F2P) gates some decks behind Scholar Pass subscription. Core gameplay decks are always free on mobile.

5. **Cross-deck FSRS sharing:** ~~Share state for "Tokyo" across decks?~~ **No.** Facts in different decks are different categories even if the surface content overlaps. Each deck tracks its own FSRS state independently.

6. **Chain theme sizing:** ~~Minimum 8 per theme?~~ **Determined during deck creation.** The Deck Master skill evaluates natural groupings during Phase 2 (Architecture) and sizes themes based on the domain's structure. No rigid minimum beyond ensuring the pool doesn't exhaust during combat.
