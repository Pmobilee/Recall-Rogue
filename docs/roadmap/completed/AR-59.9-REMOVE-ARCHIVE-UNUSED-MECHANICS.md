# AR-59.9 — Archive Unused Mechanics (v2 Spec Alignment)

## Overview

**Goal:** Align the active mechanic pool with the v2 spec. The v2 spec defines 26 Phase 1 mechanics across 6 card types. The current codebase has 31 mechanics (a mix of Phase 1 and 2 designations that doesn't match v2). Five mechanics are absent from the v2 spec and must be archived so they never appear in new runs.

**Estimated complexity:** Low — mechanical change only, no new systems.

**Dependencies:** AR-59.8 (balance rebalance) should be done first so all active mechanics already have v2 values.

## Sub-Steps

### Step 1: Mark Archived Mechanics in mechanics.ts

Extend `MechanicDefinition` with an optional `archived?: true` flag. Apply it to the 5 mechanics listed above.

Update `getActiveMechanics()` to also filter out `archived: true` mechanics:

```ts
function getActiveMechanics(): MechanicDefinition[] {
  const phase = ENABLE_PHASE2_MECHANICS
    ? MECHANIC_DEFINITIONS
    : MECHANIC_DEFINITIONS.filter(m => m.launchPhase === 1);
  return phase.filter(m => !m.archived);
}
```

Do NOT delete the five mechanic objects — they must remain in `MECHANIC_DEFINITIONS` and `MECHANIC_BY_ID` for save-data backward compatibility.

**File:** `src/data/mechanics.ts`

**Acceptance:** `getActiveMechanics()` returns exactly 26 entries. `MECHANIC_BY_ID['parry']` still resolves (for old saves).

---

### Step 2: Verify Pool Exclusion

Because `MECHANICS_BY_TYPE` is built from `getActiveMechanics()`, the five archived mechanics will automatically be excluded from card generation once Step 1 is complete.

Confirm by checking any call sites that read from `MECHANICS_BY_TYPE` or `getActiveMechanics()` — no additional changes should be needed. If any code does a direct lookup by ID to assign mechanics at run-build time (not just resolve existing cards), those sites must also exclude archived mechanics.

Search for: references to `'parry'`, `'cleanse'`, `'transmute'`, `'immunity'`, `'overclock'` as string literals used in pool-building or starter-deck logic.

**File:** Any file that builds card pools (e.g., `src/services/encounterBridge.ts`, starter deck builders).

**Acceptance:** A new run never deals a card with one of the five archived mechanic IDs.

---

### Step 3: Deprecate Archived Handlers in cardEffectResolver.ts

In `src/services/cardEffectResolver.ts`, locate the resolution branches for `parry`, `cleanse`, `transmute`, `immunity`, and `overclock`.

Add a `// @deprecated — archived mechanic, kept for save-data compatibility` JSDoc comment above each branch. Do NOT remove the logic — old saves with these cards must still resolve correctly.

**File:** `src/services/cardEffectResolver.ts`

**Acceptance:** Archived handlers are annotated. No logic is removed. TypeScript still compiles cleanly.

---

### Step 4: Update Tests

- Remove any test assertions that expect archived mechanics to appear in active pools or new-run card rewards.
- Keep resolution tests for archived mechanics (they validate save-data backward compatibility).
- If a test iterates `MECHANICS_BY_TYPE` and asserts a specific count, update the expected count to 26.

**Files:** Files under `src/` or `tests/` containing `vitest` / `describe` blocks referencing archived mechanic IDs or pool counts.

**Acceptance:** `npx vitest run` passes with no failures.
