# AR-59.20 — Echo Mechanic V2: Charge-Only Echo Cards

## Overview

**Goal:** Overhaul the Echo card mechanic so that Echo cards can ONLY be played via Charge. Quick Play is permanently blocked on Echo cards. This forces the player to retry the quiz on a fact they previously got wrong, with meaningful stakes: correct Charge redeems the fact at full power with a double FSRS stability bonus; wrong Charge plays at reduced power and exhausts the Echo card.

**Dependencies:**
- AR-59.1 (Quick Play / Charge System) — MUST be implemented first. This AR assumes `PlayMode`, CHARGE button, and Quick Play tap flow all exist.
- AR-59.11 (V2 Relic Catalogue) — required for Insight Prism integration (Step 7).

**Estimated Complexity:** Medium — touches `balance.ts`, `encounterBridge.ts`, `CardHand.svelte`, `CardCombatOverlay.svelte`, `cardEffectResolver.ts`, and `playerData.ts`.

**Source of Truth:** This document is the implementation spec.

## Correct Echo Charge

1. Card resolves at **1.0× base power** (stored `baseEffectValue` is full value after Step 2).
2. Golden flash celebration effect on the card (reuse existing animation mechanism or add `.echo-correct-flash` keyframe).
3. FSRS double stability bonus: call `applyEchoStabilityBonus(factId, ECHO.FSRS_STABILITY_BONUS_CORRECT_V2)` where `FSRS_STABILITY_BONUS_CORRECT_V2 = 6.0`.
4. Remove fact from `run.echoFactIds` (fact redeemed — cannot spawn another Echo this run).

## Wrong Echo Charge

1. Card resolves at **0.5× base power** — apply `ECHO.POWER_MULTIPLIER_WRONG` at resolve time.
2. No golden flash. Normal wrong-answer feedback.
3. FSRS records second miss (standard wrong-Charge FSRS path — `updateReviewStateByButton(factId, 'again')` — verify it fires for Echo wrong answers).
4. Echo is **exhausted** immediately: call `exhaustCard(activeDeck, playedCard.id)` (already exported from `deckManager.ts`). Card is removed from play and cannot be re-drawn this run.

## Insight Prism Relic Integration

When `insight_prism` relic is held AND the current card is an Echo (`card.isEcho`), check `run.insightPrismAutosucceedIds` (AR-59.11 run-state field) at Charge start:

- If `run.insightPrismAutosucceedIds.has(card.factId)`:
  - Quiz overlay opens, correct answer is pre-highlighted.
  - After 300ms, auto-resolves as correct without player input.
  - Remove factId from `insightPrismAutosucceedIds` (one-time use per fact per run).
  - Triggers full Correct Echo Charge path (1.0× power, golden flash, double FSRS bonus, fact redeemed).
- If factId NOT in auto-succeed set: quiz is live — no special behavior.

> Note: The Insight Prism `on_charge_wrong` trigger (AR-59.11) populates `insightPrismAutosucceedIds` when a non-Echo card is answered wrong on a Charge. The Echo spawned from that wrong answer will then auto-succeed — this is the intended Knowledge Engine synergy.


## Implementation Notes

### `echo_lens` Relic V2 Semantics

In v1, `echo_lens` restored full power because Echoes were pre-reduced at spawn. In v2, Echo correct Charge is already full power — `echo_lens` becomes redundant for correct Echoes. The relic is repurposed in Step 8: it now prevents the 0.5× wrong-Charge penalty (making wrong Echo Charges also resolve at 1.0×). This preserves `echo_lens` as a meaningful relic while keeping its flavor ("echoes at full strength").

### Coordination with AR-59.1

This AR requires `PlayMode` (from `src/data/mechanics.ts`) and the Charge/Quick Play event pipeline. Implement AR-59.1 first. If implementing both in a single pass, add `PlayMode` to `src/data/mechanics.ts` per AR-59.1 Step 1 before writing any code in this AR.

### Exhausting Echo After Wrong Charge

`exhaustCard(deck, cardId)` is already exported from `src/services/deckManager.ts` (see JSDoc comment on line ~339 noting Echo resolution as a use case). The card is removed from the hand pile and placed in the exhaust pile. It will never be reshuffled.

### `insightPrismAutosucceedIds` Run State Field

This field is defined in AR-59.11 as `insightPrismAutosucceedIds: Set<string>`. If AR-59.11 has not been implemented yet, add this field to the active run state type when implementing Step 7, or gate Step 7 behind AR-59.11 completion. Do not block Steps 1–6 on this.
