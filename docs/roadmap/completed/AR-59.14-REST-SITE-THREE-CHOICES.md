# AR-59.14 — Rest Site: Three Choices (Rest / Study / Meditate)

## Overview

**Goal:** Expand rest sites from 2 choices (Rest / Upgrade Card) to 3 choices: Rest (heal 30%), Study (quiz-to-upgrade), and Meditate (remove 1 card). The old "Upgrade Card" option is cut entirely; Study becomes the only upgrade path at rest sites.

**Dependencies:** AR-59.5 (3-act run structure with rest sites on the map)

**Estimated complexity:** Medium — 2 new inline UI state-machine flows + game-state wiring + FSRS hook-up

---

## Design Decisions

- **D20 — Study upgrades are player-choice.** After each correct answer the player is shown 3 cards from their deck and picks which one to upgrade. This matches the existing `UpgradeSelectionOverlay` pattern and preserves meaningful agency.
- **D21 — Meditate allows exactly 1 removal per rest site visit.** The shop card-removal path (50g per AR-59.15) still exists and is the repeatable option. Can't meditate if deck would drop to ≤ 5 non-Echo cards.
- **D23 — Old "Upgrade Card" path is CUT.** The `onupgrade` prop in `RestRoomOverlay.svelte`, `handleRestUpgrade()` in `CardApp.svelte`, and the `{#if $currentScreen === 'upgradeSelection'}` block (rest-only usage) are all removed. The `upgradeSelection` screen state and `UpgradeSelectionOverlay.svelte` remain alive for the post-mini-boss rest path — do NOT remove them.

---

## Three Choices

### Rest — Heal 30% max HP
- No quiz. Safe, passive option. Behavior unchanged from current implementation.
- `healAmount = Math.round(playerMaxHp * 0.3)`, capped at `playerMaxHp`.
- `data-testid="rest-heal"` (unchanged)

### Study — Answer up to 5 questions, earn card upgrades

**Player flow:**
1. Player selects "Study".
2. Up to 5 quiz questions presented one at a time, sourced from `getActiveDeckFactIds()` in `src/services/encounterBridge.ts` → look up each `factId` in `factsDB` to get a `Fact`, then render via the existing `QuizOverlay` component (`mode="study"`).
3. **Correct answer:** Show correct-answer reveal briefly (~1 s), then immediately show a pick-1-of-3 upgrade selector. Candidates sourced via `getUpgradeCandidates(deck, 3)` from `src/services/cardUpgradeService.ts`. Player taps a card to upgrade it (`upgradeCard(card)` from the same service — sets `isUpgraded: true`, appends "+" to `mechanicName`, applies stat deltas).
4. **Wrong answer:** No penalty. Show the correct answer as a learning moment (standard post-quiz reveal). Continue to next question.
5. "Done Studying" button always visible during the study flow — exits early and calls `onRestResolved()`.
6. After all 5 questions or early stop: call `onRestResolved()` and return to map.
7. FSRS update for every answer (correct and wrong): call `updateReviewStateByButton(factId, button)` from `src/ui/stores/playerData.ts`, where `button = 'good'` for correct answers (no speed bonus at rest sites) and `button = 'again'` for wrong answers.
8. Upgrade priority: `getUpgradeCandidates()` already sorts by Tier 2b > 2a > 1 — offer those top 3 as upgrade choices.

### Meditate — Remove 1 card from the run deck

**Player flow:**
1. Player selects "Meditate".
2. Full run deck displayed as a scrollable card grid. All non-Echo cards shown (`card.isEcho !== true`). Use `getActiveDeckCards()` from `src/services/encounterBridge.ts` and filter out Echo cards.
3. Player taps a card to select it for removal.
4. Inline confirmation prompt: "Remove [mechanicName] from your deck? This cannot be undone." with Confirm and Cancel buttons.
5. On confirm: call `removeCardFromActiveDeck(cardId)` (new function in `src/services/deckManager.ts`), then call `onRestResolved()`.
6. On cancel: return to card grid.
- Limited to 1 removal per rest site visit (tracked via internal `$state` boolean `hasUsedMeditate`).
- Meditate button disabled (greyed, tooltip shown) if `getActiveDeckCards().filter(c => !c.isEcho).length <= MIN_DECK_SIZE` where `MIN_DECK_SIZE = 5` (add to `src/data/balance.ts`).
- Echo cards are excluded from the removal grid entirely.

### Step 4 — Implement Study flow in `RestRoomOverlay.svelte`
**File:** `src/ui/components/RestRoomOverlay.svelte`

Internal state for Study:
```ts
let questionsAnswered = $state(0)
let currentQuestion = $state<{ fact: Fact; choices: string[] } | null>(null)
let studyUpgradeCandidates = $state<Array<{ card: Card; preview: UpgradePreview }>>([])
let showAnswerReveal = $state(false)
let lastAnswerCorrect = $state(false)
const MAX_STUDY_QUESTIONS = 5
```

**Study flow steps (within the overlay):**
1. On entering `studying` phase: fetch first question — shuffle `deckCards` to get a random `factId`, look it up via `factsDB.getById(factId)`, build `choices` array (correct answer + distractors from `fact.distractors`). Show question using `QuizOverlay` component with `mode="study"` and `onAnswer` callback.
2. In `onAnswer(correct)` callback:
   a. Call `updateReviewStateByButton(fact.id, correct ? 'good' : 'again')` from `src/ui/stores/playerData.ts`.
   b. Increment `questionsAnswered`.
   c. If correct: set `showAnswerReveal = true`, `lastAnswerCorrect = true`, prepare upgrade candidates via `getUpgradeCandidates(deckCards, 3)` from `cardUpgradeService`. After 800 ms reveal delay, transition to `upgradeSelect` phase.
   d. If wrong: set `showAnswerReveal = true`, `lastAnswerCorrect = false`. After 1200 ms reveal delay, call `loadNextQuestion()`.
3. `loadNextQuestion()`: if `questionsAnswered >= MAX_STUDY_QUESTIONS`, call `props.onstudy()` (signals parent to call `onRestResolved()`). Otherwise, load next fact from deck pool.
4. `upgradeSelect` phase: render pick-1-of-3 card grid (same visual style as `UpgradeSelectionOverlay`). On card pick: call `upgradeCard(card)` from `cardUpgradeService`, show brief "Upgraded!" flash, then call `loadNextQuestion()`.
5. "Done Studying" button: always visible in `studying` and `upgradeSelect` phases. On click: call `props.onstudy()`.

**Fact sourcing:** Shuffle `deckCards.map(c => c.factId)`, deduplicate (use a `Set<string>`), cycle through up to 5 unique facts. If the deck has fewer than 5 unique facts, stop early.

**Acceptance:**
- Correct answer → upgrade picker appears with 3 cards.
- Wrong answer → correct answer shown, no upgrade picker, next question loads.
- "Done Studying" always navigates back to map.
- FSRS updated for every answer.

### Step 5 — Implement Meditate flow in `RestRoomOverlay.svelte`
**File:** `src/ui/components/RestRoomOverlay.svelte`

Internal state for Meditate:
```ts
let selectedCardForRemoval = $state<Card | null>(null)
let hasUsedMeditate = $state(false)
```

**Meditate flow steps:**
1. On entering `meditating` phase: display `deckCards` (non-Echo only, already filtered by parent) in a scrollable 2-column grid. Same card styling as the grid in `UpgradeSelectionOverlay.svelte`.
2. Tap a card: set `selectedCardForRemoval = card`. Show inline confirmation modal layered over the grid: "Remove [card.mechanicName] from your deck? This cannot be undone." with "Remove" (danger red) and "Cancel" buttons.
3. On "Remove" confirm: emit `props.onmeditate(selectedCardForRemoval.id)`, set `hasUsedMeditate = true`, transition back to `choice` phase (so player sees the main 3-choice screen again but Meditate is now disabled for this visit).
4. On "Cancel": set `selectedCardForRemoval = null`, stay in `meditating` phase.

The Meditate button in the `choice` phase should also be disabled after `hasUsedMeditate === true` (in addition to the parent-level deck-size check).

**Acceptance:**
- Deck grid shows all non-Echo cards.
- Confirmation modal appears on card tap.
- Cancel returns to grid.
- Confirm emits removal event, returns to choice phase, Meditate now greyed.
- Cannot trigger twice in one rest visit.
