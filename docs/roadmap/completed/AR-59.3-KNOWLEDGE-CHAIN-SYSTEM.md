# AR-59.3 — Knowledge Chain System

## Overview

**Goal:** Add a Knowledge Chain mechanic: when consecutive cards played in the same turn share a `categoryL2` value, they form a chain with escalating multipliers. This REPLACES the Knowledge Combo system (consecutive correct answers across turns).

**Dependencies:**
- AR-59.1 (Charge multiplier must exist — chain stacks multiplicatively with Charge)
- AR-59.17 (Chain Visual System) handles colored tints, pulse, and connecting line VFX — out of scope here
- AR-59.19 will remove the old Knowledge Combo system that this replaces

**Estimated complexity:** Medium (new service + integration into 4 existing files + run pool modification + UI hint)

**Key constraint:** The `Card` entity does NOT currently carry `categoryL2` — it lives only on `Fact`. Step 0 must propagate `categoryL2` onto `Card` so the chain system can read it at play time without hitting the DB.

---

## Design Decisions

- **D6:** Run pool must concentrate on 5–8 active `categoryL2` groups per run. With ~50 possible values, chain probability is too low to matter. With 5–8 groups, 2-chain probability ≈ 70–80%, 3-chain ≈ 20–30%.
- **D7:** Facts shuffle per draw (current behavior — keep). Chain opportunities shift every turn naturally.
- **D8:** No cap on peak multipliers. Chain × Charge can reach 9.0×+. Tune enemy HP, not caps.
- **D9:** Chain tracks per-turn only (resets at turn start). No cross-turn chain persistence.
- **D10:** Wrong answers (fizzles) do NOT extend a chain — the card is discarded without a `categoryL2` match event. A fizzle breaks the conceptual sequence but does NOT reset the chain tracker (the chain simply does not advance).

## Chain Multiplier Table

| Chain Length | Multiplier | Visual Feedback |
|---|---|---|
| 1 (no chain) | 1.0× | Normal play |
| 2-chain | 1.3× | Subtle glow, thin connecting line (AR-59.17) |
| 3-chain | 1.7× | Bright glow, particle trail, chain sound (AR-59.17) |
| 4-chain | 2.2× | Screen edge pulse, chain lightning VFX (AR-59.17) |
| 5-chain | 3.0× | Full celebration, screen shake, "KNOWLEDGE CHAIN!" (AR-59.17) |

**Stacking examples (chain × Charge, multiplicative):**
- 3-chain Quick Play Strike: `8 × 1.0 × 1.7 = 13.6` per card
- 3-chain Charged correct Strike (Tier 2): `8 × 3.0 × 1.7 = 40.8` per card
- 5-chain all Charged on Surge: `8 × 3.0 × 3.0 = 72` per card
- With Prismatic Shard relic: `8 × 3.5 × 3.0 = 84` per card

## How Chains Work

1. At start of each player turn, `currentChain = { categoryL2: null, length: 0 }`.
2. When a card is played AND answered correctly:
   - a. Read the card's `categoryL2` value (propagated from Fact onto Card in Step 0).
   - b. If `card.categoryL2 === currentChain.categoryL2`: increment `length`, apply `CHAIN_MULTIPLIERS[length]`.
   - c. If `card.categoryL2 !== currentChain.categoryL2`: reset chain to `{ categoryL2: card.categoryL2, length: 1 }`, multiplier = 1.0×.
3. Chain multiplier is applied AFTER Charge multiplier: `finalEffect = baseEffect × playModeMultiplier × chainMultiplier`.
4. Chain resets at the start of each new turn (in `endPlayerTurn` → `drawHand` flow).
5. Fizzled cards (wrong answers) do NOT advance or reset the chain.

---

## Run Pool `categoryL2` Concentration

The run pool builder must select **5–8 active `categoryL2` groups** for the run based on the selected domains. This ensures most drawn hands contain chain opportunities.

Implementation approach:
1. After `collectDomainFacts` resolves the full candidate pool, group facts by `categoryL2`.
2. Select the top 5–8 groups by fact count (ensuring educational breadth across groups).
3. Mark these groups as "concentrated" — bias sampling to draw at least 80% of pool facts from these groups.
4. Remaining 20% can come from other `categoryL2` values (for variety and fallback coverage).
5. This is layered on top of the existing subcategory-balance logic — it replaces the `POOL_SUBCATEGORY_MAX_PCT` cap for the primary/secondary domain facts but does not affect the review-card or filler-card buckets.

### Step 2: Create `chainSystem.ts` Service

**File:** `src/services/chainSystem.ts` (NEW)

This is a pure in-memory state tracker. It is NOT a Svelte store — it is a mutable singleton reset per encounter turn. UI-reactive state is exposed separately in Step 5.

```ts
// === Chain System ===
// Tracks the Knowledge Chain state for the current player turn.
// A chain forms when consecutive correctly-answered cards in the same turn
// share the same categoryL2 value.

import { CHAIN_MULTIPLIERS, MAX_CHAIN_LENGTH } from '../data/balance';

export interface ChainState {
  /** The categoryL2 value of the current chain, or null if no chain is active. */
  categoryL2: string | null;
  /** Current chain length. 0 = no cards played yet. 1 = first card (no bonus). */
  length: number;
}

let _chain: ChainState = { categoryL2: null, length: 0 };

/**
 * Resets the chain at the start of each player turn.
 * Called by turnManager at the start of each new turn.
 */
export function resetChain(): void {
  _chain = { categoryL2: null, length: 0 };
}

/**
 * Called when a card is played correctly.
 * Extends the chain if categoryL2 matches, or resets it to a new chain.
 *
 * @param categoryL2 - The categoryL2 value of the card just played. Undefined = no chain contribution.
 * @returns The chain multiplier that applies to this card's effect.
 */
export function extendOrResetChain(categoryL2: string | undefined): number {
  if (!categoryL2) {
    // Card has no categoryL2 — reset chain, no bonus
    _chain = { categoryL2: null, length: 0 };
    return 1.0;
  }

  if (categoryL2 === _chain.categoryL2) {
    _chain.length = Math.min(_chain.length + 1, MAX_CHAIN_LENGTH);
  } else {
    _chain = { categoryL2, length: 1 };
  }

  return getChainMultiplier(_chain.length);
}

/**
 * Returns the current chain multiplier for the given chain length.
 */
export function getChainMultiplier(length: number): number {
  const clamped = Math.min(Math.max(0, length), MAX_CHAIN_LENGTH);
  return CHAIN_MULTIPLIERS[clamped] ?? 1.0;
}

/**
 * Returns the current chain length (0 if no chain active).
 */
export function getCurrentChainLength(): number {
  return _chain.length;
}

/**
 * Returns a snapshot of the current chain state for UI consumption.
 */
export function getChainState(): Readonly<ChainState> {
  return { ..._chain };
}
```

**Acceptance criteria:**
- [ ] `extendOrResetChain('mammals')` × 3 returns [1.0, 1.3, 1.7]
- [ ] `extendOrResetChain('birds')` after mammals resets length to 1, returns 1.0
- [ ] `resetChain()` zeroes out state
- [ ] Cards with no `categoryL2` return 1.0 and reset the chain
- [ ] `npm run typecheck` passes

### Step 7: Update Run Pool Builder for `categoryL2` Concentration

**File:** `src/services/runPoolBuilder.ts`

After `collectDomainFacts` resolves primary and secondary facts but before the pool is finalized, apply `categoryL2` concentration:

1. Group the combined primary+secondary facts by `categoryL2`.
2. Sort groups by size (descending).
3. Select the top N groups where `N = random(CHAIN_CATEGORYL2_GROUPS_PER_RUN.min, CHAIN_CATEGORYL2_GROUPS_PER_RUN.max)`. Use the run RNG if active.
4. Partition pool: facts in the top-N groups are "concentrated"; the rest are "peripheral".
5. Target: 80% of primary/secondary slots go to concentrated facts, 20% to peripheral.
6. This replaces the current `POOL_SUBCATEGORY_MAX_PCT` cap behavior for `categoryL2` grouping (the existing per-subcategory cap can remain but `categoryL2` concentration overrides the target distribution).

Fallback: if fewer than 5 `categoryL2` groups are available (e.g. very small domain), use all available groups — do not fail or error.

**Acceptance criteria:**
- [ ] Run pool has 5–8 distinct `categoryL2` values among its primary/secondary facts
- [ ] At least 75% of primary/secondary facts belong to the concentrated groups
- [ ] Works when domain has fewer than 5 `categoryL2` groups (uses all available)
- [ ] `npm run typecheck` passes

## Notes on AR-59.19 (Combo Removal)

The existing combo multiplier system (`COMBO_MULTIPLIERS`, `comboCount`, `COMBO_HEAL_THRESHOLD`) is NOT removed here. AR-59.19 handles that removal. Until AR-59.19 is implemented, both systems coexist — the chain multiplier stacks with the combo multiplier. This is acceptable during the transition period (balance will be re-tuned in AR-59.19 when combo is removed).
