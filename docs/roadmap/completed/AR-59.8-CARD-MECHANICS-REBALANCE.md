# AR-59.8 — Card Mechanics Rebalance (v2 Spec)

## Overview

Redefine all 25 Phase 1 card mechanics with three distinct numeric profiles: **Quick Play** (1.0× — plays immediately, no quiz), **Charged Correct** (3.0× — answered correctly), and **Charged Wrong** (0.7× — answered incorrectly). For buff/debuff mechanics, Charging gives **bonus effects** (qualitative changes) rather than just numeric scaling.

This is a pure data + resolver rebalance. No new mechanics are added. Mechanics currently marked `launchPhase: 2` in `mechanics.ts` that appear in the v2 spec table are promoted to the Phase 1 pool as part of this task.

**Dependencies:**
- AR-59.1 (Quick Play / Charge Play system) — the `playMode` field must exist on resolved card plays before this task's resolver changes will fire
- AR-59.9 (will remove/consolidate unused mechanics after this rebalance — do not delete any mechanics in this task)

**Estimated complexity:** Medium-High. Data-heavy with careful resolver branching required. Expect ~5 files changed.

## Context: Current State

### `src/data/mechanics.ts`

The `MechanicDefinition` interface currently has a single `baseValue: number` field (and optional `secondaryValue`, `secondaryThreshold`). There is no per-play-mode value split. The resolver in `cardEffectResolver.ts` reads `mechanic.baseValue` directly.

Current Phase 1 mechanics (12 total, `launchPhase: 1`):
`strike`, `multi_hit`, `block`, `thorns`, `cleanse`, `empower`, `quicken`, `weaken`, `expose`, `scout`, `recycle`, `mirror`, `adapt`, `emergency`

The v2 spec promotes the following `launchPhase: 2` mechanics to Phase 1:
`heavy_strike`, `piercing`, `reckless`, `execute`, `lifetap`, `fortify`, `brace`, `overheal`, `focus`, `double_strike`, `hex`, `slow`, `foresight`

(`cleanse`, `parry`, `transmute`, `immunity`, `overclock` are NOT in the v2 table — leave them as-is at their current `launchPhase` values; AR-59.9 will decide their fate.)

### `src/data/balance.ts`

`ENABLE_PHASE2_MECHANICS = true` is already set at line 1237. The `BALANCE` object contains combat base values (e.g., `BASE_EFFECT`) but no per-mechanic Quick Play / Charged tables. Those values currently live as `baseValue` inside each `MechanicDefinition`.

### `src/services/cardEffectResolver.ts`

The resolver calls `getMechanicDefinition(card.mechanicId)` and branches on `mechanic.id` to apply special effects. It does NOT currently branch on `playMode`. The `resolveCardEffect` function signature does not accept a `playMode` parameter.

### `src/services/cardDescriptionService.ts`

Generates card text. Currently reads `mechanic.baseValue` for numeric display. Does not show play-mode-specific values.

**Key design rules:**
1. Wrong answers always yield SOMETHING (0.7× base) — always worse than Quick Play but never zero.
2. 0-AP cards (`quicken`, `foresight`) cost **1 AP** to Charge. The quiz IS the cost.
3. Buff/debuff Charge bonuses are qualitative (new effects), not just bigger numbers.
4. `reckless` self-damage stays flat at 3 regardless of play mode — only the damage output scales.

## Sub-Steps

### Step 1: Extend `MechanicDefinition` Interface in `mechanics.ts`

Add three new optional fields to the `MechanicDefinition` interface:

```ts
export interface MechanicDefinition {
  id: string;
  name: string;
  type: CardType;
  description: string;
  baseValue: number;          // Keep for backward compat — Quick Play value
  apCost: number;
  maxPerPool: number;
  secondaryValue?: number;
  secondaryThreshold?: number;
  tags: string[];
  launchPhase: 1 | 2;

  // v2 play-mode values
  quickPlayValue?: number;           // Quick Play effect value (= baseValue for compat)
  chargeCorrectValue?: number;       // Charged Correct effect value
  chargeWrongValue?: number;         // Charged Wrong effect value
  chargeCorrectSecondary?: number;   // Secondary value for Charged Correct (e.g., thorns reflect)
  chargeWrongSecondary?: number;     // Secondary value for Charged Wrong
  /** Qualitative bonus effects granted only on Charged Correct. */
  chargeBonusEffect?: ChargeBonusEffect;
}

export type ChargeBonusEffect =
  | 'quicken_draw'          // Quicken: draw 1 extra card
  | 'focus_double'          // Focus: 2 AP discounts instead of 1
  | 'double_strike_pierce'  // Double Strike: adds Pierce flag to the buffed attack
  | 'slow_weaken'           // Slow: also applies Weaken 1t to enemy
  | 'foresight_intent'      // Foresight: reveals next enemy intent
  | 'recycle_from_discard'  // Recycle: also draws 1 card from discard
```

Then populate ALL 25 mechanic entries in `MECHANIC_DEFINITIONS` with the v2 values from the table above.

**Mechanic promotions:** Set `launchPhase: 1` for:
`heavy_strike`, `piercing`, `reckless`, `execute`, `lifetap`, `fortify`, `brace`, `overheal`, `focus`, `double_strike`, `hex`, `slow`, `foresight`

(Do NOT change `launchPhase` for `cleanse`, `parry`, `transmute`, `immunity`, `overclock`.)

**Acceptance criterion:**
- TypeScript interface compiles with no errors
- Every mechanic in the v2 table has non-undefined `quickPlayValue`, `chargeCorrectValue`, `chargeWrongValue`
- Mechanics with bonus effects have `chargeBonusEffect` set
- `npm run typecheck` passes

### Step 2: Add `playMode` to Resolver Signature in `cardEffectResolver.ts`

The resolver must know which mode a card was played in. Add `playMode` to `AdvancedResolveOptions`:

```ts
export interface AdvancedResolveOptions {
  activeRelicIds?: Set<string>;
  isFirstAttackThisEncounter?: boolean;
  isDoubleStrikeActive?: boolean;
  isFocusActive?: boolean;
  isOverclockActive?: boolean;
  damageDealtThisTurn?: number;
  /** How this card was played. Defaults to 'quick' if not provided. */
  playMode?: 'quick' | 'charge_correct' | 'charge_wrong';
}
```

Then update `resolveCardEffect` to read `playMode` from `advanced` and select the correct value columns:

```ts
const playMode = advanced.playMode ?? 'quick';

function getMechanicValue(mechanic: MechanicDefinition | undefined, fallback: number): number {
  if (!mechanic) return fallback;
  switch (playMode) {
    case 'charge_correct': return mechanic.chargeCorrectValue ?? mechanic.baseValue * 3;
    case 'charge_wrong':   return mechanic.chargeWrongValue   ?? mechanic.baseValue * 0.7;
    default:               return mechanic.quickPlayValue     ?? mechanic.baseValue;
  }
}
```

For each mechanic branch in the resolver, replace direct `mechanic.baseValue` reads with `getMechanicValue(mechanic, fallback)`. Also handle secondary values with an equivalent helper.

**Bonus effect branching:** For each `chargeBonusEffect` type, add conditional logic:
- `quicken_draw`: when `playMode === 'charge_correct'`, set `result.extraCardsDrawn += 1`
- `focus_double`: when `playMode === 'charge_correct'`, set `result.focusCharges = 2` instead of 1
- `double_strike_pierce`: when `playMode === 'charge_correct'`, add a `damageDealtBypassesBlock: true` flag on the buffed card (or set a new `result.doubleStrikeAddsPierce = true` flag for the turn manager to apply)
- `slow_weaken`: when `playMode === 'charge_correct'`, also apply Weaken 1t status to enemy
- `foresight_intent`: when `playMode === 'charge_correct'`, set `result.applyForesight = true` AND `result.revealNextIntent = true`
- `recycle_from_discard`: when `playMode === 'charge_correct'`, set `result.drawFromDiscard = 1`

**0-AP Charge cost:** `quicken` and `foresight` have `apCost: 0`. When `playMode !== 'quick'`, the charge system (AR-59.1) is responsible for deducting 1 AP. The resolver does NOT need to handle this — it is purely the play-system's job. Document this assumption in a comment.

**Acceptance criterion:**
- `resolveCardEffect` accepts `playMode` in `AdvancedResolveOptions`
- Quick Play resolves to `quickPlayValue` (≈ current behavior for cards already tuned)
- Charged Correct resolves to `chargeCorrectValue` (3× base for most mechanics)
- Charged Wrong resolves to `chargeWrongValue` (0.7× base)
- Bonus effects fire only on `charge_correct`
- `npm run typecheck` passes

### Step 3: Update Card Descriptions in `cardDescriptionService.ts`

Card text must reflect what the player will actually get. The service already has a `getShortCardDescription` and `getCardDescriptionParts` API used by `CardHand.svelte`.

**Acceptance criterion:**

## Notes for Implementor

- `reckless` self-damage: the self-damage value (3) should be stored as a separate flat constant, NOT scaled. Only the `chargeCorrectValue` / `quickPlayValue` / `chargeWrongValue` apply to the outgoing damage.
- `brace` has `baseValue: 0` currently because it scales from enemy intent at runtime. The new `quickPlayValue: 1.0` means "1× enemy intent multiplier", `chargeCorrectValue: 3.0` means "3×", `chargeWrongValue: 0.7` means "0.7×". The resolver's brace branch should use `getMechanicValue(mechanic, 1.0)` as the multiplier against enemy intent.
- The `CardEffectResult` interface will need new optional fields: `revealNextIntent?: boolean`, `drawFromDiscard?: number`, `doubleStrikeAddsPierce?: boolean`. Add these at the top of `cardEffectResolver.ts` alongside the existing fields.
- Do NOT delete `cleanse`, `parry`, `transmute`, `immunity`, or `overclock` from `MECHANIC_DEFINITIONS`. AR-59.9 handles mechanic pruning.
