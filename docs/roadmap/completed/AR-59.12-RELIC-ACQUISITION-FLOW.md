## Overview

**Goal:** Define and implement exactly when and how relics are offered during a run. This AR replaces the current starter relic selection screen and the ad-hoc drop logic scattered across `gameFlowController.ts` with a single, coherent acquisition flow grounded in the three-act structure.

**Core principle:** Runs start with zero relics. The first relic is earned by defeating the Act 1 mini-boss (~encounter 8-10 of a full run). The opening fights — before the mini-boss — are intentionally relic-free, making that first reward feel earned rather than handed out.

**What this AR does NOT do:**
- Does not change the five-slot cap, sell/swap mechanics, or `RelicSwapOverlay` (AR-59.10)
- Does not define which relics exist in the catalogue (AR-59.11)
- Does not change map generation or act structure (AR-59.5)
- Does not visually redesign `RelicRewardScreen` or `RelicPickupToast` (minor prop additions only)

**What this AR completes:**
- Removes the `starterRelicSelection` screen and `onStarterRelicSelected` function (builds on AR-59.10 Step 6; completes it if not yet landed)
- Fixes `generateRandomRelicDrop` to use full rarity weights instead of common/uncommon-only
- Updates `RELIC_BOSS_RARITY_WEIGHTS` values to match design spec (common 0.20, rare 0.30)
- Adds a pity timer (`relicPityCounter`) to `RunState`
- Wires elite node completions to the choose-1-of-3 relic reward flow
- Centralizes all acquisition paths through a consistent `resolveRelicAcquisition(trigger)` dispatcher
- Changes `offeredRelicIds` scope so declined relics can reappear
- Adds reroll button wiring to `RelicRewardScreen`
- Documents the complete trigger table in `docs/GAME_DESIGN.md`

## Acquisition Trigger Table

| Trigger | Award Type | Options | Rarity Weights | UI |
|---------|-----------|---------|----------------|----|
| Act 1 mini-boss (first mini-boss of run) | Choose 1 of 3 | 3 | Regular | `RelicRewardScreen` |
| Subsequent mini-bosses | 1 random drop | 1 | Regular | `RelicPickupToast` |
| Elite node kill | Choose 1 of 3 | 3 | Regular | `RelicRewardScreen` |
| Boss node kill | Choose 1 of 3 | 3 | Boss | `RelicRewardScreen` |
| Regular combat (10% chance) | 1 random drop | 1 | Regular | `RelicPickupToast` |

**Regular weights:** Common 50%, Uncommon 30%, Rare 15%, Legendary 5%
**Boss weights:** Common 20%, Uncommon 35%, Rare 30%, Legendary 15%

## Pity Timer

After 4 consecutive relic acquisitions that are all Common rarity, the next drop is **guaranteed Uncommon or higher**, regardless of the standard weight roll.

**State field added to `RunState`:**

```typescript
/** Consecutive common-only relic drops since last Uncommon+ acquisition. Resets on Uncommon+. */
relicPityCounter: number;
```

**Logic:**
- Increment `relicPityCounter` by 1 whenever a Common relic is acquired (any path).
- Reset `relicPityCounter` to 0 whenever an Uncommon, Rare, or Legendary relic is acquired (any path).
- When `relicPityCounter >= RELIC_PITY_THRESHOLD` (4) at the time of a draw, force a re-roll excluding Common. Cap attempts at 10 to avoid infinite loops; fall back to the full pool if no Uncommon+ relics remain in the pool.
- Pity applies to both toast drops and `RelicRewardScreen` choice sets. For a choose-1-of-3 event, if pity is active, at least one of the three options must be Uncommon+.

## Pool Exclusions

The `buildRelicPool()` function in `gameFlowController.ts` excludes:
1. Relics already held (`run.runRelics` — existing behavior)
2. Relics not in `save.unlockedRelicIds` or `STARTER_RELIC_IDS` (existing behavior)
3. After AR-59.11: relics with `excludeFromPool === true` (unbuilt mechanics)
4. After AR-59.10: relics requiring Mastery Coin purchase the player hasn't completed

**Declined relics CAN reappear.** Currently `offeredRelicIds` on `RunState` blocks reappearance of every shown relic. This AR changes that behavior: `offeredRelicIds` is used only to prevent the same relic appearing twice within a **single reroll session**. It is scoped to the active selection event and cleared when the event resolves. Declined relics are not permanently blacklisted.

> Design rationale: In Slay the Spire, declined relics can come back. It creates meaningful decisions — "I passed on this before, but now my build needs it."

## Relic Reroll

At boss, mini-boss (first), and elite relic selections (choose-1-of-3 events):
- **Cost:** 50g (constant `RELIC_REROLL_COST` from AR-59.10)
- **Limit:** Once per selection event
- **Behavior:** All 3 current options are replaced with 3 fresh draws. The replaced options enter a temporary `rerollSeenIds` set (scoped to this event only), preventing them reappearing during this reroll.
- **UI:** "Reroll (50g)" button on `RelicRewardScreen`. Disabled when gold < 50 or already rerolled this event.

AR-59.10 Step 9 specifies the backend functions (`rerollRelicSelection`, `canRerollRelicSelection`, `resetRelicSelectionRerolls`). This AR wires those to `RelicRewardScreen` via new props.

## No Starter Relic

- Run starts with `run.runRelics = []` (0 relics).
- First 2-4 encounters (Act 1 regular combats, before the mini-boss) are relic-free by design.
- First relic opportunity: Act 1 mini-boss (approximately encounter 8-10 depending on path taken).
- The `starterRelicSelection` screen state is removed from the game flow state machine. (AR-59.10 Step 6 specifies this; this AR completes it if that step hasn't landed.)

## Integration with Five-Slot System (AR-59.10)

When any relic is about to be awarded, the acquisition path checks slot capacity before adding:

```
resolve trigger
  → build pool
  → draw options (1 or 3 depending on trigger type)
  → apply pity if needed
  → if choose-1-of-3 (RelicRewardScreen):
      player picks one
      → isRelicSlotsFull?
          YES → trigger RelicSwapOverlay (AR-59.10 Step 8/11)
          NO  → addRelicToRun, updateRelicPityCounter
  → if random drop (RelicPickupToast):
      → isRelicSlotsFull?
          YES → show toast WITH "Swap" button (see Step 10 below)
               dismissing without swapping = relic lost (intentional — low stakes)
          NO  → addRelicToRun, show toast, updateRelicPityCounter
```

### Step 3 — Fix `generateRandomRelicDrop` to Use Full Weighted Draw

**File**: `src/services/relicAcquisitionService.ts`

Replace the current `generateRandomRelicDrop` implementation (which hard-filters to `common | uncommon` only) with a full weighted draw that respects the pity flag:

```typescript
/**
 * Picks a single random relic from the pool using the provided rarity weights.
 * If pityActive is true, forces the selection to Uncommon or higher rarity.
 * Falls back to the full pool if no Uncommon+ relics are available (prevents deadlock).
 * Returns null if the pool is empty.
 */
export function generateRandomRelicDrop(
  pool: RelicDefinition[],
  rarityWeights: Record<RelicRarity, number>,
  pityActive: boolean = false,
): RelicDefinition | null {
  if (pool.length === 0) return null;

  let eligible = pool;
  if (pityActive) {
    const uncommonPlus = pool.filter(r => r.rarity !== 'common');
    if (uncommonPlus.length > 0) {
      eligible = uncommonPlus;
    }
    // If no Uncommon+ available, fall through to full pool
  }

  return generateRelicChoices(eligible, 1, rarityWeights)[0] ?? null;
}
```

Update all call sites in `gameFlowController.ts` to pass `RELIC_RARITY_WEIGHTS` and the pity flag from `isRelicPityActive()` (defined in Step 4).

**Acceptance criteria**: Random drops can now return Rare and Legendary relics. Pity flag correctly forces Uncommon+. `npm run typecheck` passes.

### Step 4 — Add Pity Counter Helpers to `gameFlowController.ts`

```typescript
/** Returns true when the pity counter has reached the threshold for Uncommon+ guarantee. */
function isRelicPityActive(): boolean {
  const run = get(activeRunState);
  return (run?.relicPityCounter ?? 0) >= RELIC_PITY_THRESHOLD;
}

/** Update the pity counter after any relic acquisition. Call inside addRelicToRun. */
function updateRelicPityCounter(relic: RelicDefinition): void {
  const run = get(activeRunState);
  if (!run) return;
  if (relic.rarity === 'common') {
    run.relicPityCounter = (run.relicPityCounter ?? 0) + 1;
  } else {
    run.relicPityCounter = 0;
  }
  activeRunState.set(run);
}
```

## Design Notes & Open Questions

**Q: Should elite rewards use boss rarity weights or regular weights?**

Regular weights for elites. Elites are harder than regular combats but below mini-boss tier. If playtesting shows elite relics feel underwhelming, a "medium" weight table (common 35%, uncommon 35%, rare 20%, legendary 10%) can be added in a future balance AR rather than over-engineering it now.

**Q: Does pity apply to choose-1-of-3 events?**

Yes, minimally. If pity is active, at least one of the 3 options must be Uncommon+. In practice, with regular weights (50% Common, 30% Uncommon, 15% Rare, 5% Legendary), the odds of all 3 being Common is only 12.5%. Pity is a backstop, not a guarantee of good options.

**Q: Do shop relic purchases reset or advance the pity counter?**

No. Shop relics are economic decisions, not drops. Pity tracks passive drops from encounters only (toast drops and choose-1-of-3 acquisitions). Buying a Common from the shop does not increment the counter.

**Q: What if `offeredRelicIds` no longer blocks reappearance — will the same relic appear repeatedly?**

Only held relics are permanently excluded. Since most runs have 16-18 encounters and fewer than 10-12 relic events, and the pool has 38-42 eligible relics, the practical reappearance rate of any specific relic is low. The design matches STS behavior. If this causes UX issues in playtesting (same relic appearing 3x per run), a soft "recently seen" cooldown can be added in a follow-up balance AR.

**Q: What does `offeredRelicIds` track after this AR?**

Only acquired relics. It serves as an anti-duplicate guard: if a relic is in `offeredRelicIds`, it means the player already holds it (or held it earlier and sold it — in which case it could be offered again, which is acceptable).
