# AR-59.2 — CHARGE! Gesture UX

## Overview
**Goal:** Add the CHARGE! button and fling-up gesture as two complementary ways to trigger Charge Play. This is the UX layer built on top of AR-59.1's Quick Play / Charge Play system.

**Dependencies:**
- AR-59.1 (Quick Play / Charge Play system) — must be implemented first. This doc assumes `onChargePlay(cardIndex: number)` is exposed by `CardCombatOverlay.svelte` and passed down to `CardHand.svelte`.
- AR-59.4 (Surge turns) — will update button cost badge from "+1" to "+0"; no changes needed here to support that.

**Estimated complexity:** Medium (2 files, CSS-heavy, gesture threshold logic already partially in place)

---

## Design Decisions

- **D1: Two-tap flow is preserved.** Tap to inspect → tap popped card = Quick Play. The CHARGE! button and fling gesture are additive paths, not replacements.
- **D2: CHARGE! button is a separate DOM element** floating above the popped/inspected card. It is never reachable by accidental tap on the card itself.
- **D3: Fling gesture reuses the existing drag infrastructure** in `CardHand.svelte`. The existing thresholds (`isDragPreview` at 40px, `isDragPastThreshold` at 60px, `handlePointerUp` cast at >60px) are extended to a two-stage model with a new 80px hard-commit threshold.
- **D4: Tier 3 (Mastered) cards** suppress the CHARGE! button entirely — those cards auto-resolve at 1.2× on Quick Play with no quiz interaction needed.
- **D5: Insufficient AP suppresses the button** with a greyed-out state. The affordance remains visible so players understand the system exists.

### Step 2 — Update Drag Thresholds

Replace the existing single 60px threshold with a two-stage model:

| Variable | Old value | New value | Meaning |
|---|---|---|---|
| `isDragPreview` | `dragDeltaY > 40` | unchanged | golden glow begins |
| `isDragPastThreshold` | `dragDeltaY > 60` | `dragDeltaY > 60` | intermediate (still used for CSS class `drag-ready`) |
| `isDragChargeReady` | — | `dragDeltaY > 80` | green pulse, release triggers Charge |

In `handlePointerUp`:
- Old: `deltaY > 60` → `oncastdirect(index)` (Quick Play via fling)
- New:
  - `deltaY > 80` AND `onchargeplay` defined → call `onchargeplay(index)` (Charge Play)
  - `deltaY > 80` AND `onchargeplay` NOT defined → fall back to `oncastdirect(index)` (graceful degradation until AR-59.1 lands)
  - `deltaY <= 80` with drag movement → snap back (no action)
  - `!wasDrag` → `onselectcard(index)` (unchanged tap-to-inspect)

The existing `wasDrag` check (`Math.abs(deltaY) > 20 || deltaX > 20`) is preserved to distinguish taps from drags.

**Files:** `src/ui/components/CardHand.svelte`

**Acceptance criteria:**
- Fling-and-release below 80px → card snaps back, no action
- Fling-and-release above 80px → `onchargeplay` fires (or `oncastdirect` if not yet wired)
- Tap (movement <20px total) still calls `onselectcard` as before

### Step 3 — CHARGE! Button Rendering

Inside the `{#each cards as card, i}` block in the template, add the CHARGE! button as a sibling element AFTER the `<button class="card-in-hand">` block. Render it conditionally:

**Show when ALL of the following are true:**
1. `selectedIndex === i` (card is in popped/inspected state)
2. `card.tier !== '3'` (Tier 3 auto-resolves, no button needed)
3. `onchargeplay` is defined (AR-59.1 is wired up)
4. `cardPlayStage` is not `'committed'` (don't show during active quiz)

**Button structure:**

```html
{#if selectedIndex === i && card.tier !== '3' && onchargeplay}
  {@const chargeApAffordable = (card.apCost ?? 1) + 1 <= apCurrent || isSurgeActive}
  <button
    class="charge-play-btn"
    class:charge-btn-disabled={!chargeApAffordable}
    disabled={!chargeApAffordable}
    title={!chargeApAffordable ? 'Not enough AP' : 'Charge Play — answer for bonus damage'}
    onclick={() => onchargeplay!(i)}
  >
    CHARGE!
    <span class="charge-ap-badge">{isSurgeActive ? '+0' : '+1'} AP</span>
  </button>
{/if}
```

### Step 6 — Prevent Accidental Charges

The following safeguards must all be in place:

1. **Tap vs drag discrimination** — existing logic: `wasDrag = Math.abs(deltaY) > 20 || deltaX > 20`. Taps with <20px total movement will NEVER reach the charge branch.

2. **Minimum drag distance** — 80px (deliberate gesture on mobile). Existing 60px threshold was already intentional; 80px is stricter.

3. **CHARGE! button is a separate element** — can never be hit by a tap on the card face. No double-tap risk.

4. **Post-play debounce** — after `onchargeplay` or `oncastdirect` fires, `CardCombatOverlay.svelte` sets `cardPlayStage = 'committed'`. While committed, the CHARGE! button is hidden (see Step 3 condition). This prevents double-play.

5. **`disabled` state on `CardHand`** — when `disabled={true}` is passed (e.g., during enemy turn), `handlePointerDown` returns early. No gesture processing occurs.

No new debounce timer is needed — the existing `cardPlayStage` flow already prevents double-play.

**Files:** no new code required; verify existing guards are intact after Steps 2–5

**Acceptance criteria:**
- Rapid taps on a card never trigger Charge
- Tapping CHARGE! button while `cardPlayStage === 'committed'` does nothing (button is hidden)
- Dragging 79px and releasing → snap back, no action
