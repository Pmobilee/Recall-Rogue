## Attempt A: 2-Peer Docker Lockstep

**Result: not-attempted (infrastructure blocker)**

The multiplayer transport (`src/services/multiplayerTransport.ts`) uses BroadcastChannel, a browser same-origin API. Messages only propagate within a single browser context — they cannot cross separate Docker container Chromium instances. Spinning up two containers named `BATCH-ULTRA-t9-peer-a` and `BATCH-ULTRA-t9-peer-b` would produce two fully isolated game instances with no connectivity between them.

Additionally, the multiplayer lobby infrastructure requires both clients to join the same BroadcastChannel by lobby code, which in the simulation layer is generated by `multiplayerLobbyService.ts` and passed via the same BroadcastChannel. There is no WebSocket bridge or shared in-process message bus in the current headless test infrastructure.

**Downgrade reason documented in manifest-entry.json.**

## Top 3 Findings

### Finding 1 — CRITICAL: `crit_lens` + `obsidian_dice` co-op desync
Both relics use `Math.random()` directly in the damage computation path. In co-op mode, both clients compute damage to the shared enemy independently and submit an `EnemyTurnDelta` to the host barrier. If the two clients roll different outcomes for these relics on the same card play, their deltas will contain different `damageDealt` values — the merged result will be the arithmetic sum of two independent dice rolls rather than one agreed-upon result. The effect is stochastic damage inflation or deflation vs what either player saw locally.

**Severity: CRITICAL** — affects every co-op game where either player holds crit_lens or obsidian_dice.

### Finding 2 — PASS (regression confirmed): 3 recent RNG fixes hold under re-test
All three gotchas from 2026-04-09 (`enemyIntents` fork, `enemyPool` fork, map `assignEnemyIds` local rng) are confirmed fixed. The Bug 6 isolation test (consuming global enemies fork before map gen) produces identical enemy ID assignments — the exact scenario that previously caused co-op desync is now blocked.

### Finding 3 — INFRASTRUCTURE: 2-peer headless test not feasible with current transport
The BroadcastChannel transport is inherently same-browser — no true 2-peer headless lockstep test can be run without either a WebSocket bridge (partially scaffolded in `multiplayerTransport.ts`) or an in-process shared message bus for testing. This is a gap in the test infrastructure, not a bug in the game.

## Creative Pass

### 1. "While I was in there..."
Noticed `randomUtils.ts` exports a single `shuffled<T>()` function using `Math.random()` with no RNG parameter. This is used in `nonCombatQuizSelector.ts` (multiple sites) and `bountyManager.ts`. The quiz selector uses it for distractor shuffling (low co-op impact), but the bounty manager (`bountyManager.ts:117`) uses it to shuffle bounty candidates — and in a seeded run that shuffle is non-deterministic. This is Green-zone to note (not fix — fixing it would touch bountyManager.ts outside scope). Logged in manifest.

### 2. "A senior dev would..."
The current co-op reconciliation model (optimistic local + end-of-turn delta merge) is correct for deterministic damage, but the two remaining `Math.random()` calls in `relicEffectResolver` reveal a deeper architectural tension: the "pure function" contract for card effects breaks when relics introduce per-client randomness. A senior dev would add a `relicRng?: SeededRng` parameter to `resolveChargeEffect` (or its inner helper that handles the relic multiplier chain), threaded from the call site. In co-op mode the host could pre-roll and include the relic RNG outcomes in the delta payload — effectively making the delta the canonical truth rather than a reconstruction. This is fundamentally cleaner than seeding both clients to match.

### 3. "Player would want..."
The `obsidian_dice` relic's 50/50 roll is intended to be exciting and visible — players want to see the dice roll animation and know the outcome. In co-op, even after fixing the desync, both players need to see the SAME roll outcome — not two independent ones. This argues for broadcasting the roll result (not just the damage delta) so both screens can show the same "crit!" or "miss!" feedback simultaneously. The fix and the UX are aligned: same seeded outcome means same visual feedback.
